# Usability Testing Plan — Alkemio Redesign Prototype

> **Purpose**: Validate the redesigned UI with internal team and key external users before committing to development  
> **Prototype**: High-fidelity Figma Make prototype (30+ pages, clickable flows)  
> **Facilitators**: Jeroen Nijkamp (UX Designer) + Lead Product Owner  
> **Date**: February 2026  
> **Status**: Draft

---

## 1. What We're Testing (and What We're Not)

This is a **UI redesign** — the underlying UX flows are intentionally the same. We're testing whether the new visual design helps, hinders, or overwhelms users who already know the platform.

**Core questions we need answered:**

1. **Findability** — Can users still find everything they need in the new UI? (Same locations, new look)
2. **Workflow continuity** — Do existing key workflows feel better, worse, or the same?
3. **Change tolerance** — Is the visual change overwhelming, or does their existing knowledge transfer?
4. **First impressions** — Does it feel professional, trustworthy, and like "an upgrade"?

**We are NOT testing:**
- New features that don't exist yet (messaging, analytics)
- Technical performance or load times
- Accessibility compliance (separate audit)
- Content strategy or copy

## 2. Two Phases

| | Phase 1: Internal | Phase 2: External |
|---|---|---|
| **Who** | 6–8 team members | 6–10 key users |
| **Purpose** | Catch obvious issues, refine test script | Validate with real users |
| **Format** | Live, moderated (in-person or video call) | Live, moderated (in-person preferred, remote if needed) |
| **When** | First | After internal fixes are applied |
| **Duration** | 1–2 weeks | 2–3 weeks (scheduling dependent) |

**Why internal first:** Your team will catch prototype dead-ends, confusing flows, and missing links before external users hit them. This protects the relationship with key users — they should get a polished experience.

---

## 3. Participants

### Phase 1: Internal (6–8 people)

Recruit a mix:

| Type | Count | Why |
|------|-------|-----|
| Platform-familiar (daily users) | 3–4 | They'll notice what moved, what's missing, what's harder |
| Less-familiar (occasional/new) | 2–3 | They'll reveal whether the new UI is learnable without legacy knowledge |
| Non-design/non-dev | At least 2 | Avoids "designer eye" bias — test with ops, sales, leadership |

Skip the developers who built the prototype — they can't unsee the decisions they made.

### Phase 2: External (6–10 people)

| Persona | Count | Selection criteria |
|---------|-------|--------------------|
| Facilitator | 3–4 | Most active Space managers; use the platform weekly+ |
| Contributor | 2–3 | Regular posters/commenters; navigate across Spaces |
| Portfolio Owner | 1–2 | If available; lower priority than Facilitator/Contributor |

Pick users who are **active enough** to have muscle memory with the current UI — that's the real test.

## 4. Key Flows to Test

Each flow becomes a **task** in the test session. Give the user a scenario, not instructions.

| # | Flow | Scenario (say this to the tester) |
|---|------|-----------------------------------|
| T1 | Posting in a Space | "You're a member of the Climate Action NL Space. You want to share an update about a new report you found. Go ahead." |
| T2 | Responding to a post | "You see a post from a colleague asking for feedback. Write a response." |
| T3 | Creating a Space | "Your organization wants to start a new initiative on Digital Inclusion. Set it up." |
| T4 | Setting up a Space | "You've created the Space. Now add a description, upload a relevant document to the Knowledge Base, and invite the first content." |
| T5 | Onboarding into a Space | "You've received an invitation to join a new Space. Walk me through how you'd get started." |
| T6 | Creating a Subspace | "Inside your Space, you want to create a focused working group. Set up a Subspace." |
| T7 | Setting up a Subspace | "Populate this Subspace — add context, upload a document to the Knowledge Base." |
| T8 | Creating a whiteboard post | "You want to brainstorm visually with your team. Create a whiteboard." |
| T9 | Creating a memo post | "Write a memo summarizing last week's progress for your team." |
| T10 | Moving between Spaces | "You're working across three Spaces today. Show me how you'd switch between them quickly." |

> **Note**: More flows will be added as internal team feedback comes in. This list is the starting set.

**Important**: Don't say "click the button in the top right" — say "share an update." Watch where they go. The scenario tests findability; instructions test obedience.

## 5. Session Structure (45–60 minutes per participant)

```
[0:00 – 0:05]  Welcome & consent
[0:05 – 0:10]  Warm-up: first impressions
[0:10 – 0:45]  Task scenarios (5–7 tasks per session)
[0:45 – 0:55]  Post-task questionnaire + open discussion
[0:55 – 1:00]  Wrap-up & thank you
```

### A. Welcome & Consent (5 min)

Say something like:

> "We're testing the design, not you. There are no wrong answers. Some things might not work yet — that's fine, just tell me what you'd expect to happen. Think out loud — say what you're looking at, what you're thinking, what confuses you. I might stay quiet sometimes — that's not because you're doing it wrong, it's because I want to see what you'd do naturally."

- Get verbal consent to record (screen + audio)
- Confirm they understand it's a prototype, not the live product

### B. First Impressions (5 min)

Show the homepage of the prototype. Don't give any task. Ask:

1. "What's your first reaction?"
2. "What do you notice first?"
3. "Does anything feel familiar? Does anything surprise you?"

This captures the **change tolerance** question before task performance muddles it.

### C. Task Scenarios (35 min)

Run 5–7 tasks per session (not all 10 — rotate tasks across participants to cover everything without exhausting anyone).

**For each task:**
1. Read the scenario aloud
2. Shut up — let them work. Resist the urge to help.
3. If they're stuck for 30+ seconds: "What are you looking for?" (not "it's over here")
4. If they hit a prototype dead-end: "What would you expect to happen here?" — note it, then guide them past it
5. After they complete (or abandon): "How did that feel compared to how you do it today?"

**Record for each task:**

| Metric | How to capture |
|--------|---------------|
| Completed? | Yes / No / Needed help |
| Time | Rough (fast / moderate / struggled) — don't use a visible stopwatch |
| Path | Where did they go first? Any wrong turns? |
| Errors | Clicked wrong thing, went to wrong page, confused by label |
| Comments | Verbatim quotes — especially emotional reactions |

### D. Post-Task Questionnaire (10 min)

After all tasks, ask these questions (verbally or on paper):

**Quantitative** (1–5 scale):

1. "Overall, how easy was the new design to use?" (1 = very difficult, 5 = very easy)
2. "How overwhelming did the change feel?" (1 = very overwhelming, 5 = not at all)
3. "How confident are you that you could find everything you need?" (1 = not at all, 5 = very confident)
4. "Compared to the current design, the new design is..." (1 = much worse, 3 = same, 5 = much better)
5. "How professional and trustworthy does it feel?" (1 = not at all, 5 = very much)

**Qualitative** (open-ended):

6. "What's the one thing you like most about the new design?"
7. "What's the one thing that frustrated you or felt wrong?"
8. "Is there anything you expected to find but couldn't?"
9. "Would you be comfortable switching to this tomorrow? Why or why not?"

### E. Wrap-up (5 min)

- Thank them
- Ask if there's anything else on their mind
- For external users: "We may reach out for a short follow-up — is that OK?"

## 6. What to Look For (Analysis Framework)

After each session, log findings using three categories:

### 🔴 Blockers
Things that stopped the user from completing a task. These must be fixed.
- *Example: "User couldn't find how to create a Subspace — looked in the wrong tab for 2 minutes and gave up"*

### 🟡 Friction
Things that slowed the user down or caused confusion, but they recovered.
- *Example: "User hesitated at the Knowledge Base tab — said 'I think it used to be called something else?'"*

### 🟢 Wins
Things that were noticeably better than the current design — or that users praised.
- *Example: "User said 'Oh, this is way cleaner' when they saw the Space home page"*

### Patterns to watch for

| Signal | What it means |
|--------|---------------|
| Multiple users stuck on the same flow | Navigation/IA issue — the design made something harder to find |
| Users say "I like it" but struggle on tasks | Visual appeal ≠ usability. Dig into the task performance data. |
| "Where did [X] go?" | Something moved that shouldn't have, or the new label is unclear |
| Users complete tasks faster but feel less confident | The new UI may feel unfamiliar even when it works — that's a training/onboarding issue, not a design issue |
| First impression is positive + tasks go smoothly | Ship it. |

---

## 7. Between Phase 1 and Phase 2

After internal testing, before external:

1. **Triage findings** — Sort into Blockers / Friction / Wins
2. **Fix blockers** in the prototype (broken links, missing pages, confusing labels)
3. **Decide on friction items** — fix now or note for development phase
4. **Refine the test script** — Did any scenario confuse testers? Was the wording unclear? Adjust.
5. **Update the task list** — Add any new flows identified by the internal team

This gap should be **3–5 working days** — enough to fix the prototype, not so long that momentum dies.

## 8. Practical Tips (First-Timer's Guide)

Since this is your first moderated usability test outside of university, here are the things that matter most:

**Before the session:**
- Do a dry run with a colleague who isn't a test participant. Run the full script. Fix awkward wording.
- Test the recording setup. Screen + audio. Figma prototype in the browser, not the Figma app (easier to screen-share).
- Print the task scenarios so you're not reading off your screen while they're sharing theirs.

**During the session:**
- **The hardest skill is silence.** When the user pauses, wait. Count to 10 in your head. They'll usually figure it out or tell you what's wrong. If you jump in, you lose data.
- **Never say "you can find that in the top menu."** Say "where would you look for that?" Their answer is the finding.
- **Don't defend the design.** If they say "I don't like this," ask "tell me more" — not "well, the idea was..."
- **Take notes on paper**, not on screen. Typing is distracting. Or: have the Product Owner take notes while you facilitate.

**Role split:**
- **Jeroen (facilitator)**: Runs the session, reads scenarios, asks follow-ups
- **Product Owner (note-taker)**: Records observations, timestamps interesting moments, captures verbatim quotes

**After each session:**
- Spend 10 minutes immediately after debriefing with the Product Owner: "What stood out? Any surprises?"
- Log findings in a shared sheet while they're fresh — don't batch this to the end of the week

## 9. Logistics Checklist

### For each session:

- [ ] Prototype link works and is up to date
- [ ] Recording set up and tested (screen + audio)
- [ ] Task scenarios printed / accessible
- [ ] Observation template ready (task × metrics grid)
- [ ] Quiet room / stable video connection
- [ ] Consent confirmed (verbal or written)

### For remote sessions:

- [ ] Use screen-sharing (Zoom, Meet, Teams) — participant shares their screen
- [ ] Send the prototype link 5 minutes before, not days before (avoid pre-exploring)
- [ ] Ask them to think aloud more than usual (you can't see their face as clearly)

### For external users:

- [ ] Schedule 60-min slots with 15-min buffers (sessions always run over)
- [ ] Send a brief, friendly intro email: "We're testing a redesign and would love your feedback — 45–60 minutes of your time"
- [ ] Consider a small thank-you (gift card, Alkemio swag, heartfelt email)

---

## 10. Deliverable: Findings Report

After both phases, produce a single document:

```
1. Executive Summary (1 page)
   - What we tested, with whom, key findings
   
2. Scorecard
   - Quantitative ratings (averaged across participants)
   - Task completion rates
   
3. Key Findings
   - Blockers (with screenshots, quotes)
   - Friction points (with frequency — how many users hit this?)
   - Wins (with quotes)
   
4. Recommendations
   - Must fix before development
   - Should fix during development
   - Nice to have / future iteration
   
5. Raw Data (appendix)
   - Per-participant task logs
   - Verbatim quotes
   - Questionnaire scores
```

This report becomes the **go/no-go artifact** — the thing the team reviews to decide "yes, build this" or "fix these things first."
