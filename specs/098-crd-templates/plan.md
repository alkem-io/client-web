## Summary

Re-implement the entire Templates system on the CRD design system (shadcn/ui + Tailwind), behind the existing `alkemio-crd-enabled` toggle (default OFF), following the parallel-design-system pattern used by 039 / 041 / 042 / 043 / 045 / 096. The Apollo data layer is reused as-is; thin **data mappers** under `src/main/crdPages/` bridge GraphQL shapes to plain-TS CRD component props.

**Keystone**: a holder-agnostic **`TemplatesManagerView`** — a holder's templates grouped into one section per type (Space, Collaboration tools / Callout, Whiteboard, Post, Community Guidelines), with per-type Create / Import / Preview / Edit / Duplicate / Delete actions — used by **both** template holders (a Space's templates set in Space Settings → Templates; an Innovation Pack's templates set in the pack admin) and (read-only) on a pack's public profile. Action availability **mirrors the legacy page-access gate, not a per-type privilege**: in any *management* context (Space Settings → Templates, the pack admin) all per-type Create / Edit / Delete / Duplicate are enabled for all five types and Preview always; in a *read-only* context (a pack's public profile) only Preview. The one intentional holder-kind difference is that **Import-from-library is offered for a Space holder only** (a pack is a template source, not a sink). The legacy holder quirks are removed (packs no longer jump straight to Edit; **and there is no read-only "imported/platform-origin" template state — within a management context every template in a set is editable/duplicable/deletable**, correcting the early CRD Space-Settings "custom only" cut, which never existed in the legacy `TemplatesAdmin`). The CRD per-type `can*` predicates therefore resolve to a uniform `true` (management) or `false` (read-only) — no new authorization machinery.

Around the keystone: the reusable **`TemplatePicker`** (two modes — *import-into-Space* = persistent library manager, Space-only; *consumption* = single-pick-then-apply — with a free-text search filter over name/description/tags mirroring the legacy import dialog; source sections **Account** + **Platform** always, **Space** only when applicable — omitted in import-into-that-same-Space mode and where there's no Space context; empty sections omitted; preview before selection); the **per-type preview** (callout framing & contribution defaults — every kind, incl. **Collabora-document** title/placeholder and **poll** question/options, read-only — / whiteboard drawing / post description / space phases & starter callouts / guidelines title + text + references); the **per-type create/edit forms** — the **Callout template form being the heavyweight one, which reuses the existing CRD callout-authoring components** (`FramingEditorConnector` and the per-kind framing connectors — `WhiteboardFramingConnector`, `MemoFramingConnector`, `CollaboraFramingConnector`/`CollaboraFramingEditorOverlay`, `MediaGalleryFramingConnector`, `CallToActionFramingConnector` — plus the callout-settings controls and `ResponseDefaultsConnector`/`ResponseDefaultsDialog`, and `calloutFormMapper.ts`; **the framing-kind type is the existing CRD callout-layer `FramingChip` union — `'none'|'whiteboard'|'memo'|'document'|'cta'|'image'|'poll'`, defined in `src/crd/forms/callout/types.ts` — reused (re-exported by `crd-templates`'s `types.ts`), not a parallel `FramingKind`**), not built from scratch; the Whiteboard form reuses `WhiteboardEditorShell` + the CRD preview-settings/crop dialogs (`PreviewSettingsDialog`/`PreviewCropDialog`); the Post form reuses `MarkdownEditor`; the Space form is authored by copying an existing space/subspace; the Community Guidelines form is a guidelines-**title** field + `MarkdownEditor` (body) + a references-list editor; the **`SetDefaultTemplateDialog`** with two `purpose`s having **different source scopes** — `defaultSubspaceTemplate` picks from the **Space's own Space templates only** (via `updateTemplateDefault`; the default must reference a template in the set — the legacy selector is a plain dropdown of those, plus a preview here as an enhancement), and `flowStateDefaultCalloutTemplate` picks through the full Space/Account/Platform `TemplatePicker` (via `setDefaultCalloutTemplateOnInnovationFlowState` — the legacy "Default post template" mislabel is corrected to "Default callout template"); the **Innovation Pack** admin (pack-details form + the templates-manager view) and **public profile** page; the **Innovation Library** page (`/innovation-library`); the **CRD community-guidelines editor** delivered to legacy parity — guidelines title + reference links + markdown body, working — superseding `045`'s markdown-only stub (FR-038), as the host for the guidelines apply/save-as flows; the **picker wiring** into the consumption hosts; and the **"save as a template"** flows (US9) — a callout (no-regression) / the current community guidelines (a deliberate new addition — legacy only *applies* a guidelines template) / a subspace (no-regression) → opens the matching **Create Template** dialog (`TemplateFormDialog` in create mode) **pre-filled** from that entity. New i18n namespace: `crd-templates`. **Delivered as one PR** (the `tasks.md` phases are internal build-order milestones).

### Open verifications (do these first; they gate dependent UI)

These are **additive `.graphql` *fragment* edits** (no new documents) followed by `pnpm codegen` + a committed schema diff (Constitution III). The relevant schema fields/enums already exist (`CalloutFramingType` has `CollaboraDocument` and `Poll`; `CommunityGuidelines` has a profile displayName/references), so each is expected to be a fragment-only change — **but the very first Phase-2 task verifies each field exists server-side on the type the fragment reaches; anything that needs a real backend change is flagged as a blocker before the dependent UI is built.** V1, V2 and V5 are now *confirmed gaps* (decided in the 2026-05-12 post-analyze interview); V3 is conditional; V4 is a confirm-no-gap.

- **V1 — Collabora-document framing on `CalloutTemplateContent`.** The fragment fetches `framing.{type, whiteboard{content}, link, memo, mediaGallery}` — no Collabora-document field. A Callout template must store Collabora-document framing (spec FR-020/FR-030). Add the field to `TemplateContent.graphql` + codegen + schema diff. *(Confirmed gap.)*
- **V2 — Poll framing on `CalloutTemplateContent`.** The `CalloutFramingType` enum has `Poll`; the fragment doesn't fetch poll framing. **In scope** (spec D2, FR-020/FR-030): add the poll-framing field to `TemplateContent.graphql` + codegen + schema diff so a Callout template can carry poll framing content; the form's `FramingChip` picker already has `'poll'` and now persists. *(Confirmed gap — was "decide", now "implement".)*
- **V3 — delete-cascade data (FR-019).** Verify whether `deleteTemplate` server-side nulls the referencing `TemplateDefault` (and the referencing innovation-flow-state default callout template). If the client must clear them, verify it can learn *which* defaults / flow-states reference template X from the queries it already has; if not, add the field(s). *(Conditional.)*
- **V4 — Innovation Library card fields + search.** Verify `useInnovationLibraryQuery` exposes everything the CRD gallery card needs (template display name, description, tags, visual, type, owning-pack name + url) without a follow-up fetch — the legacy gallery uses `TemplateCardProfileInfo`, which it does; confirm no gap. Also confirm whether the legacy Library page has a free-text search box and, if so, add it (FR-007/FR-051).
- **V5 — `title` on `CommunityGuidelinesTemplateContent`.** The CG template content + form carry a distinct `title` (the guidelines' display name) in addition to the markdown body and references (spec D4, FR-020/FR-038). Verify the fragment exposes `profile.displayName` for the guidelines (it likely already does — the CG content fetch reads `profile`); if not, add it. Also confirm the live community-guidelines query exposes title + references for the editor host (FR-038). *(Likely already present — verify.)*

**Structure Decision.** Presentational CRD components live under `src/crd/components/{templates,innovationPack,innovationLibrary}/`; the `045`-era files under `src/crd/components/space/settings/` are refactored-in-place (the section view becomes a thin wrapper; the dialogs are removed/moved/generalised). All data fetching, mapping, mutation wiring, routing, toggle-gating, and the CRD↔MUI bridge replacements live under `src/main/crdPages/{templates,innovationPack,innovationLibrary}/` and the existing `src/main/crdPages/topLevelPages/spaceSettings/{templates,subspaces,layout,community}/`. Legacy `src/domain/{templates,templates-manager,InnovationPack,spaceAdmin/SpaceAdminTemplates}/` and `src/main/topLevelPages/InnovationLibraryPage/` stay intact and serve toggle-off.

**Per-type-form placement (CRD-layer-rules compliance).** `TemplateFormDialog` is a pure `src/crd/` shell: common fields (name/description/tags/banner) + a `perTypeFormSlot: ReactNode` (the ShareDialog `shareOnAlkemioSlot` pattern) + Save/Cancel/discard-guard. It does **not** import any callout/whiteboard *connector*. The per-type forms split by whether they need Apollo/`@/domain/*`: the **Post**, **Community Guidelines**, **Whiteboard** (via `WhiteboardEditorShell`), and the **presentational Space-source URL-paste picker** forms are pure → they live in `src/crd/components/templates/forms/`; the **Callout** form composes the integration-layer callout-authoring connectors (`FramingEditorConnector`, the per-kind framing connectors incl. Collabora, the callout-settings *controls*, `ResponseDefaultsConnector`, `calloutFormMapper.ts`) → it lives in **`src/main/crdPages/templates/CalloutTemplateForm.tsx`** (the integration layer, which may import Apollo/`@/domain/*`). The Space form's URL-resolver + space-content + Update-privilege Apollo plumbing likewise lives in the integration layer (folded into `useTemplateForms`), mirroring the legacy MUI `SpaceContentFromSpaceUrlForm` UX: the user pastes a space URL, clicks "Use this space", and the hook resolves the URL → fetches the space's innovation flow + callout titles → verifies the user has Update privilege on the resolved space → sets `sourceSpaceId` and the preview. `useTemplateForms` assembles the right slot per type and hands it to `TemplateFormDialog`. **Consequence for sequencing**: because the Callout connector form will import `CalloutSettingsConnector` (or its settings controls), the `CalloutSettingsConnector` → legacy-MUI-`CreateTemplateDialog` bridge must be removed **in Phase 2** (before that form is built), otherwise the CRD templates bundle transitively pulls MUI in — violating FR-002 / SC-004; tasks.md re-sequences this (T069's MUI-removal half → a Phase-2 task; its CRD-`SaveAsTemplateDialog`-wiring half stays in Phase 9).

**Holder-agnostic contract** — `TemplatesManagerView` takes `holderKind: 'space' | 'innovationPack'`, `categories`, `loading`, `duplicatingId`/`deletingId`, the `canCreate/canEdit/canDelete/canImport(type)` predicates, and `onCreate/onImport/onTemplateAction(id, action)`. It renders nothing about navigation or Apollo and has **no `isCustom` gate** (the legacy `TemplatesAdmin` has none). Built by `useTemplatesManager`: in a *management* context (Space Settings → Templates, pack admin) `canCreate/canEdit/canDelete` resolve uniformly `true` for all types and `canImport` is `true` only for `holderKind === 'space'`; in a *read-only* context (the pack public profile) all four are `false` and `onTemplateAction` is limited to `'preview'`. This matches the legacy page-access gate exactly — no per-type privilege check is introduced.

**Picker contract** — `TemplatePicker` with a discriminated `mode`: `'import'` (persistent library manager, Space-only — gallery, import individually, mark/remove `alreadyInSet`, dialog stays open; **the Space source section is omitted in this mode** — a set is not a source for importing into itself; Account + Platform only) or `'select'` (single-pick-then-apply + "no template" clear; Space source shown when there is a Space context and the target sink is something other than that set). Both share a **free-text search filter** (name/description/tags — mirroring the legacy `ImportTemplatesDialogGallery`), the Account/Platform (and, when applicable, Space) source sections (empty omitted; "no templates available" when all empty), the lazy preview pane, and `TemplatePickerCard`. Consumption hooks instantiate `'select'`; only the Space Settings tab (and the Space-holder branch of `useTemplatesManager`) instantiate `'import'`.

**"Save as a template"** — not a separate dialog: it opens `TemplateFormDialog` in `intent: 'create'` with `value` pre-filled from the source entity (the Callout connector form populated from the callout — `crdPages/templates/CalloutTemplateForm`; a Community Guidelines form populated from the current guidelines; a Space form with `sourceSpaceId` = the subspace). If the source has unsaved changes, a `ConfirmationDialog` ("save or discard first") gates it. `useSaveAsTemplate` builds the pre-filled `value`, picks the `perTypeFormSlot`, and routes the submit.
