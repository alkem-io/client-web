# Phase 0 Research — CRD Templates System

**Refreshed 2026-05-12** to match plan rev. 4 (post-`/speckit.analyze` interview, decisions D1–D13 — see `assumptions.md` / spec.md "Clarifications") and a code-level check of the legacy MUI implementation. This feature has **no open `[NEEDS CLARIFICATION]` markers** in the spec. Phase 0 covers (a) whether the existing data layer covers every CRD surface and where it doesn't (the **open verifications V1/V2/V3/V4/V5** below — all additive `.graphql` fragment edits, server-side-verified first), (b) the patterns for re-skinning the heavy legacy logic without MUI, (c) the legacy MUI realities the CRD work must match (page-access gating, the minimal pack-create dialog, read-only provider, the route layout, the two different "set default" source scopes), (d) the CRD community-guidelines editor this feature delivers to legacy parity (FR-038), and (e) the seam with the in-flight `045`/`041` work. Each item is a decision with rationale and alternatives weighed.

## 1. Does the existing GraphQL cover everything the CRD surfaces need? — *mostly yes; one change is likely (Collabora framing on `CalloutTemplateContent`); confirm V1–V4 in-PR*

**Decision**: Reuse the existing template/pack/library Apollo operations as-is wherever possible. Canonical set:
- Holder templates list: `AllTemplatesInTemplatesSet` (`src/domain/templates/components/TemplatesAdmin/AllTemplatesInTemplatesSet.graphql`) — used for both a Space's set (`space.templatesManager.templatesSet`) and a pack's set (`innovationPack.templatesSet`).
- Space holder wiring: `SpaceTemplatesManager` (`src/domain/space/graphql/SpaceTemplatesQueries.graphql`) → `templatesManager.id` + `templatesSet.id` + `templateDefaults`. Default-subspace-template selector source: `SpaceContentTemplatesOnSpace` (`src/domain/templates-manager/SelectDefaultSpaceTemplate/SpaceContentTemplatesOnSpace.graphql`) — the Space's own `spaceTemplates` only.
- Template content (per type, conditional fetches): `TemplateContent` (+ fragments `CalloutTemplateContent` / `WhiteboardTemplateContent` / `CommunityGuidelinesTemplateContent` / `SpaceTemplateContent`) and the `SpaceTemplateContent($spaceId)` query (preview a space before capturing it) — `src/domain/templates/graphql/TemplateContent.graphql` — via `useTemplateContentLazyQuery`.
- Picker sources: `ImportTemplateDialog` / `ImportTemplateDialogAccountTemplates` / `ImportTemplateDialogPlatformTemplates` (`src/domain/templates/components/Dialogs/ImportTemplateDialog/ImportTemplateDialog.graphql`).
- Mutations: `CreateTemplate`, `CreateTemplateFromSpace`, `CreateTemplateFromContentSpace`, `UpdateTemplate`, `UpdateCalloutTemplate`, `UpdateTemplateFromSpace`, `DeleteTemplate` (`TemplateMutations.graphql`); `UpdateCommunityGuidelines`, `UpdateWhiteboard` (used by legacy `TemplatesAdmin`); `updateTemplateDefault` (`TemplatesManagerMutations.graphql`); `setDefaultCalloutTemplateOnInnovationFlowState` / `removeDefaultCalloutTemplateOnInnovationFlowState`; `useCreateCalloutTemplate` (the callout-as-template helper).
- Innovation Library page: `InnovationLibrary` (`src/main/topLevelPages/InnovationLibraryPage/InnovationLibraryPage.graphql`) → `platform.library.templates` (filterable by `types`) + `platform.library.innovationPacks`.
- Innovation Pack admin: `AdminInnovationPack` (the `AdminInnovationPackPage.tsx` query) + `UpdateInnovationPack` + `DeleteInnovationPack`; pack **creation** is the `createInnovationPack` mutation — generated hook **`useCreateInnovationPackMutation`** (NOT `useCreateInnovationPackOnAccountMutation`), input type `CreateInnovationPackOnAccountInput`, sent as `{accountID, profileData:{displayName, description}}` (name + description only). Platform-admin pack list: `PlatformAdminInnovationPacks` (`AdminInnovationPacksPage.tsx`) — not migrated by this feature.
- Innovation Pack public profile: `InnovationPackProfilePage` (`src/domain/InnovationPack/InnovationPackProfilePage/InnovationPackProfile.graphql`).
- Image uploads: `useHandlePreviewImages` (`src/domain/templates/utils/useHandlePreviewImages.ts`) and `useUploadMediaGalleryVisuals` (`src/domain/collaboration/mediaGallery/useUploadMediaGalleryVisuals`).

**Open verifications (do these first; they gate dependent UI) — plan rev. 4 §"Open verifications"; all are additive `.graphql` *fragment* edits + `pnpm codegen` + a committed schema diff (Constitution III), with the field verified server-side first (a real backend change is a BLOCKER, not a workaround):**
- **V1 — Collabora-document framing on `CalloutTemplateContent`** *(likely a change is needed)*. The fragment fetches `framing.{type, whiteboard{content}, link, memo, mediaGallery}` — no Collabora-document field. `CalloutFramingType` (schema enum) includes `CollaboraDocument`. A Callout template must be able to store Collabora-document framing (FR-020). If the field is missing on the type, add it to `TemplateContent.graphql` + `pnpm codegen` + committed schema diff (Constitution III).
- **V2 — Poll framing on `CalloutTemplateContent`** *(confirmed in scope — decision D2)*. `CalloutFramingType` has `Poll`; the fragment doesn't fetch poll framing. Add the poll-framing field so a Callout template can carry poll framing content (the `FramingChip` picker's `'poll'` now persists; the preview renders it read-only). FR-020/FR-030.
- **V3 — delete-cascade data (FR-019)**. Verify whether `deleteTemplate` server-side nulls referencing `TemplateDefault`s and the referencing flow-state default callout template. If the client must clear them, verify it can learn *which* defaults / flow-states reference template X from existing queries; add a field via codegen if not.
- **V4 — Innovation Library card fields + search**. Verify `useInnovationLibraryQuery` exposes everything the CRD gallery card needs (template displayName, description, tags, visual, type, owning-pack name + url) without a follow-up fetch (the legacy gallery uses `TemplateCardProfileInfo`, which it does). Also: the legacy import dialog's `ImportTemplatesDialogGallery` has a per-section free-text search box (over displayName/description/tags) — the CRD picker reproduces it (FR-016); confirm whether the legacy Library page (`LibraryTemplatesView`) also has one and, if so, add it there too (FR-007).
- **V5 — `title` on `CommunityGuidelinesTemplateContent` + the live CG query (decision D4 / FR-020 / FR-038)**. The CG template content/form carry a distinct `title` (the guidelines' `profile.displayName`) in addition to the body and references; verify the template-content fragment exposes it (it likely already fetches `profile`). Also confirm the **live** community-guidelines query (the one the CRD editor host — FR-038 — uses) exposes `profile.{displayName, description, references}`; add anything missing. *(Likely already present — confirm; gates the CG `title` field and the CG editor host.)*

**Rationale**: the spec mandates "data layer unchanged" *as a default*; the legacy code already exercises almost every operation this feature needs; the one likely change (V1) follows the prescribed codegen-in-PR + schema-diff workflow, which Constitution III explicitly permits.

**Alternatives considered**: writing leaner CRD-specific queries (rejected — duplicates fragments, risks cache-shape divergence, and the legacy queries are already tuned for these screens).

## 2. Re-skinning the heavy legacy logic (preview renderers, callout-template form) without MUI — *reuse the existing CRD callout/whiteboard layer; lift only the shaping logic*

**Decision**: For each per-type **preview**, lift the *content-shaping logic* from the legacy `Template*Preview` components (`src/domain/templates/components/Previews/*`) into the integration-layer `templateContentMapper.ts`, which produces a plain-TS `TemplateContent` discriminated union; the CRD `preview/*` components are pure renderers of that union (Tailwind chrome, `lucide-react`, `MarkdownContent`/`InlineMarkdown` for markdown). The **callout** preview reuses the CRD callout-preview components (`CalloutWhiteboardPreview`, `CalloutMemoPreview`, `CalloutCollaboraPreview`, `CalloutMediaGalleryCarousel`, `CalloutLinkAction`); the whiteboard preview reuses the CRD read-only whiteboard viewer; the space-template preview reuses CRD innovation-flow phase chips + a callout list.

For the **Callout template create/edit form** (the heavyweight one), do **not** build a bespoke editor — it is a thin shell over the existing CRD callout-authoring connectors: a framing-kind picker → the matching per-kind framing connector (`WhiteboardFramingConnector` / `MemoFramingConnector` / `CollaboraFramingConnector` (+`CollaboraFramingEditorOverlay`) / `MediaGalleryFramingConnector` / `CallToActionFramingConnector` / the poll framing connector if one exists, else a minimal poll editor / plain text), the contribution-type & comment-settings *controls* (the control components used inside `CalloutSettingsConnector` — not the whole connector module, which carries the legacy-MUI `CreateTemplateDialog` bridge), the contribution defaults via `ResponseDefaultsConnector`/`ResponseDefaultsDialog`, and `calloutFormMapper.ts` for the form↔mutation mapping. It covers **every** framing kind incl. **Collabora-document** (V1) and **poll** (V2 — poll content persists). **The framing-kind type is the existing CRD callout-layer `FramingChip` union (`'none'|'whiteboard'|'memo'|'document'|'cta'|'image'|'poll'`, defined in `src/crd/forms/callout/types.ts`)** — `types.ts` re-exports it rather than redefining a parallel `FramingKind`. **Placement (CRD layer rules)**: those callout-authoring connectors import `@apollo/client` / `@/domain/*` (and, until the bridge is removed, MUI), so the file that composes them — `CalloutTemplateForm.tsx` — cannot live under `src/crd/`; it lives in the **integration layer** (`src/main/crdPages/templates/`), and the pure `src/crd/` shell `TemplateFormDialog` receives it via a `perTypeFormSlot: ReactNode` (the `ShareDialog` `shareOnAlkemioSlot` pattern). The bridge in `CalloutSettingsConnector` (its legacy-MUI `CreateTemplateDialog` import + the `CRD_SAVE_AS_TEMPLATE_ENABLED` kill-switch) is removed in **Phase 2** (`useSaveAsTemplate` re-points the menu item to the CRD dialog) — before any CRD callout-template code reaches that connector — so no CRD templates bundle pulls MUI in (FR-002 / SC-004). The **Whiteboard form** reuses `WhiteboardEditorShell` for the drawing **plus the CRD preview-settings/crop dialogs `PreviewSettingsDialog`/`PreviewCropDialog`** (FR-020 — "the whiteboard drawing and its preview settings", not just an image upload — all `src/crd/` components, so the form stays pure; if `PreviewSettingsDialog` turns out Apollo-bound, its wiring moves to the integration layer, or fall back to image-only); the Post form reuses `MarkdownEditor`; the **Community Guidelines form** is a guidelines-**title** field [required] + `MarkdownEditor` (body) + a references-list editor (the title + references sub-components are shared with the CRD community-guidelines editor host this feature also delivers — see §8 / FR-038); the Space form is a *presentational* "pick a space/subspace to copy" picker (results + `onSearch`/`onSelect` as props — the Apollo search lives in the integration layer) + recursive toggle + the `SpaceTemplatePreview` of the captured structure. Form state lives in the integration layer (`useTemplateForms`, reusing the MUI-free `Forms/common/mappings.ts`); no Formik inside `src/crd/`.

**Rationale**: the genuinely complex part is *what to show / what to edit* (the content shapes and the framing editors), which the CRD callout/whiteboard layer already implements; re-implementing it would be a large duplication and a drift risk (DRY). Keeping the shaping in the mapper makes the CRD renderers trivially testable with plain fixtures.

**Alternatives considered**: wrapping the legacy MUI preview/form components in an iframe / portal (rejected — drags MUI/Emotion into CRD routes); a bespoke template-only callout editor (rejected — duplicates `FramingEditorConnector` + the per-kind connectors + `CalloutSettingsConnector` + `ResponseDefaultsConnector`); re-deriving content shapes from raw GraphQL inside the CRD components (rejected — business logic in `src/crd/`, Constitution I/IV).

## 3. The picker's two modes vs. the legacy import dialog — *consolidate into one component with a discriminated `mode`*

**Decision**: One `TemplatePicker` component, `mode: 'import' | 'select'`. `mode: 'import'` is a **persistent library manager** — a *consolidation* of the two legacy import surfaces, not a one-to-one re-implementation of either: (a) the legacy `ImportTemplatesDialog` as used by `TemplatesAdmin` for the Space templates set actually closes after each import, shows only the **Account** + **Platform** sections (`disableSpaceTemplates: true` — a set isn't a source for importing into itself), and has *no* "already in set" marking or remove-from-set; (b) the legacy `ImportTemplatesDialog` as used by the innovation-flow default-template selector keeps the dialog open, shows the current selection at the top with a **Remove** affordance. The CRD `'import'` mode combines (a)'s Account+Platform-only scope with (b)'s stays-open + mark/remove behaviour, so several imports can be done in one session and templates already in the set can be removed from there — a deliberate improvement, signalled as such in the spec. `mode: 'select'` is what every *consumption*/default flow needs: pick exactly one template (filtered to the allowed type(s)), preview it, confirm → close, the consumer applies it; a persistent "no template / start blank" affordance clears the selection; the **Space** source section is shown when there is a Space context (it's a valid source for, e.g., a callout-creation flow). Both modes share a **free-text search filter** over name/description/tags (mirroring the legacy `ImportTemplatesDialogGallery`), the source-loading hook (`useImportTemplateDialog*Query`), the preview-content lazy query, and `TemplatePickerCard`.

**Rationale**: the spec's FR-016/FR-017 split these two behaviours; one component with a `mode` discriminator (ISP-friendly) avoids duplicating the source-section layout, the search box, the preview pane, and the empty-state handling. The legacy code is two behaviours bolted onto one dialog; CRD makes the split explicit and consolidates the two import variants.

**Alternatives considered**: two separate components (`TemplateImportManager` + `TemplateSelectDialog`) sharing a sub-component (rejected — more files, ~80% shared surface; a `mode` union is the cleaner OCP/ISP shape); reproducing the legacy import-into-Space dialog exactly (rejected — it closes after one import and has no remove-from-set; the consolidated behaviour is better and the spec calls it out as an intentional change); reusing the legacy MUI dialog for the import case and only building select (rejected — MUI in CRD).

## 4. Whiteboard editing/viewing inside template forms & previews — *reuse the existing CRD whiteboard editor/viewer*

**Decision**: The Whiteboard template form embeds the existing CRD whiteboard editor (the same one used elsewhere in CRD callout/whiteboard work); the Whiteboard template preview and the Callout template's "default whiteboard" preview embed the CRD read-only whiteboard viewer. No new whiteboard component. Preview images for whiteboard templates are uploaded via `useHandlePreviewImages` from the integration layer.

**Rationale**: the CRD whiteboard editor already exists and handles the Excalidraw stack; re-creating it would be a large, unnecessary duplication. The CLAUDE.md "reuse, don't re-create" rule applies directly.

**Alternatives considered**: a read-only image-only preview (rejected — regression vs. the legacy interactive whiteboard preview, and the editor is already available).

## 5. Holder unification — *one `holderKind` prop; action predicates mirror the legacy page-access gate (not a per-type privilege)*

**Decision**: `TemplatesManagerView` is holder-agnostic. It takes `holderKind: 'space' | 'innovationPack'` (used only for the few cosmetic differences a designer might want, e.g. a section subtitle) plus the per-type predicates `canCreate/canEdit/canDelete/canImport(type)`. The legacy reality the predicates must mirror: **neither `SpaceAdminTemplatesPage` nor `AdminInnovationPackPage` does any per-type authorisation** — both hard-code "all five types creatable/editable/deletable" (the Space additionally passes `canImportTemplates` all-true); the real gate is page access (being allowed to reach the surface), and a read-only context (`InnovationPackProfilePage` rendering `TemplatesAdmin` with no predicates) offers none. So the CRD consumers build the predicates as a uniform `() => true` in a management context (`canImport` only for `holderKind === 'space'`) and `() => false` in the read-only context — no new authorisation machinery, no `isCustom` gate (the legacy `TemplatesAdmin` has none — the early CRD Space-Settings "custom only" cut is a deviation, dropped here). Clicking a template **always** opens the Preview dialog first (with an Edit affordance when `canEdit(type)`), removing the legacy `alwaysEditTemplate` "packs jump straight to Edit". All Create / Edit / Preview / Delete / Duplicate go through the *same* CRD dialog components regardless of holder. The Innovation Pack admin builds the props with `holderKind='innovationPack'`; the Space Settings tab with `holderKind='space'`; the pack public profile with all-false predicates + `onTemplateAction` limited to `'preview'`.

**Rationale**: implements the spec's unification requirement (FR-012/FR-013/FR-014/FR-070) *as the legacy app actually gates it* — page access, not per type — so "no new roles or privileges" holds literally; keeps a single source of truth (DRY); a future third holder kind needs zero changes to the view (OCP).

**Alternatives considered**: introducing a genuine per-type privilege model (rejected — that would be a *new* gate, contradicting FR-070); keeping the `045` `isCustom` Edit/Delete restriction (rejected — never existed in the legacy `TemplatesAdmin`; the resolved clarification drops it); separate `SpaceTemplatesView` / `PackTemplatesView` (rejected — duplication + drift); a prop-less view deriving everything from a context (rejected — couples the CRD component to app state).

## 6. Routing & feature-toggle gating for the new pages — *mirror the established CRD route pattern; get the route files right*

**Decision** (corrected against the actual route layout): Add toggle-gated CRD routes the same way 042/045/096 did, with the lazy-import + conditional-`<Route>` pattern. The exact mount points:
- **`/innovation-library`** — in `src/main/routing/TopLevelRoutes.tsx`: a lazy `CrdInnovationLibraryPage` vs the lazy `MuiInnovationLibraryPage` (`src/main/topLevelPages/InnovationLibraryPage/`), chosen by `crdEnabled`.
- **The Innovation Pack public profile (`<pack.profile.url>`) AND the pack admin (`<pack.profile.url>/settings`)** are *both* routed from **`src/domain/InnovationPack/InnovationPackRoute.tsx`** (it has `:innovationPackNameId` → `InnovationPackProfilePage`, `:innovationPackNameId/settings` → `AdminInnovationPackPage`, plus the `/:templateNameId` deep-link variants), and `InnovationPackRoute` is mounted from `TopLevelRoutes.tsx`. So toggle-gate **`InnovationPackRoute.tsx`** (or swap `InnovationPackRoute` in `TopLevelRoutes.tsx` for a CRD variant): render `CrdInnovationPackProfilePage` / `CrdInnovationPackAdminPage` when `useCrdEnabled()` is true, else the legacy pages. **Not** `src/domain/platformAdmin/domain/innovationPacks/AdminInnovationPackRoutes.tsx` — that file only routes the platform-admin pack *list* at `/admin/innovation-packs` (`AdminInnovationPacksPage`), which is **not migrated by this feature** (an FR-037 transitional state; a platform admin opens a pack from that list and lands on the CRD admin screen when the toggle is on).
- **Pack create** is from `ContributorAccountView.tsx` (the shared user/org "Account" tab) — open `CrdCreateInnovationPackDialog` (name + description) when CRD is on, else the legacy dialog; on success navigate to `<new pack url>/settings`. **Pack delete** is the three-dot menu on each pack card in that same Account-tab packs list (the legacy app deletes account-owned packs there too — `useAccountEntityDeletion.ts`) → `ConfirmationDialog` → `useDeleteInnovationPackMutation`; **not** on the pack admin screen.
- CRD pack/library pages are wrapped in `CrdLayoutWrapper`; legacy pages keep their shells; the pack public profile and `/innovation-library` must be reachable anonymously (no identity gate) — verify the existing mounts aren't behind one.

**Rationale**: the standard migration mechanism (migration-guide §"Wire the Route (with Feature Toggle)"); getting the route file right matters because the earlier draft pointed at `AdminInnovationPackRoutes.tsx`, which is the wrong file.

**Alternatives considered**: a single route component that internally swaps shells (rejected — project standardised on lazy-import + conditional `<Route>`); shipping CRD pack/library pages unconditionally (rejected — opt-in until validated); migrating the platform-admin pack list too (out of scope here — FR-037 fallback).

## 7. "Save as template" and "set as default" — *generalise the two `045` dialogs; respect the two different "set default" source scopes*

**Decision**: `SaveSubspaceAsTemplateDialog` (045) → generalised `SaveAsTemplateDialog` (CRD `templates/`) = `TemplateFormDialog` in create mode opened **pre-filled** from a source entity: a **callout** (no-regression — `useCreateCalloutTemplate`; replaces the CRD→MUI bridge in `CalloutSettingsConnector`), the **current community guidelines** (a deliberate **new** flow — the legacy guidelines editor only *applies* a template; `CreateTemplate` with `communityGuidelinesData`), a **subspace** (no-regression — `useCreateTemplateFromSpaceMutation` / `…FromContentSpaceMutation`). `ChangeDefaultSubspaceTemplateDialog` (045) → generalised `SetDefaultTemplateDialog` (CRD `templates/`) with `purpose: 'defaultSubspaceTemplate' | 'flowStateDefaultCalloutTemplate'` — and the two purposes have **different source scopes** (matching legacy):
- `defaultSubspaceTemplate` — the candidates are the **Space's own Space templates only** (`SpaceContentTemplatesOnSpace` / the legacy `SelectDefaultSpaceTemplateDialog` is a plain `FormikSelect` over those — a default must reference a template that is *in* the set; `updateTemplateDefault(templateID)` enforces this). CRD adds a candidate preview (an enhancement); it does **not** add the Account/Platform sources here.
- `flowStateDefaultCalloutTemplate` — the candidates are **Space / Account / Platform** (the legacy `SetDefaultTemplateDialog` inside the innovation-flow editor uses the full `ImportTemplatesDialog` with `selectedTemplateId` + `onRemoveTemplate`). CRD hosts `TemplatePicker` (`mode: 'select'`) with all three sources; mutations `setDefaultCalloutTemplateOnInnovationFlowState` / `removeDefaultCalloutTemplateOnInnovationFlowState`; the label is corrected from the legacy mislabel "Default post template" to "Default callout template".

**Rationale**: these are the same interaction shape (pick/capture → use for X) with a different "X"; generalising avoids near-identical dialogs and keeps the picker the single selection surface — but the two "set default" purposes genuinely differ in *which templates are eligible*, so the dialog branches on `purpose`.

**Alternatives considered**: one `SetDefaultTemplateDialog` that always hosts the 3-source picker (rejected — wrong for `defaultSubspaceTemplate`, where only the holder's own Space templates can be the default); keeping the 045 dialogs space-specific and adding parallel ones (rejected — duplication); folding "set default" into the picker itself (rejected — "what the selection is for" belongs to the consumer hook).

## 8. Seam with spec `045-crd-space-settings` and spec `041-account-templates-dialog` — *098 owns the components; 045 hosts two of its consumers; 041 is realised by the picker*

**Decision** (recorded in the spec's Clarifications):
- `098` **owns** the Templates tab body and all reusable template machinery. `045`'s Templates tab is reduced to: render `<TemplatesManagerView holderKind="space" .../>` fed by `crdPages/templates/useTemplatesManager` (+ the default-subspace-template card calling `useSetDefaultTemplate`). The `045`-era files `SpaceSettingsTemplatesView.tsx` / `TemplateLibraryDialog.tsx` / `TemplateEditDialog.tsx` / `TemplatePreviewDialog.tsx` / `SaveSubspaceAsTemplateDialog.tsx` / `ChangeDefaultSubspaceTemplateDialog.tsx` are refactored-in-place (relocated/replaced) — **not** duplicated.
- `045`'s **Layout tab** per-phase "default callout template" and **Subspaces tab** "default subspace template" + "save subspace as template" affordances stay in `045`'s tasks but consume `098`'s `TemplatePicker` / `SetDefaultTemplateDialog` / `SaveAsTemplateDialog`.
- `045`'s **Community tab** community-guidelines *editor* is markdown-only and incomplete (no title, no references, possibly non-functional). `098` **delivers the CRD community-guidelines editor to legacy parity** — guidelines title + reference links + markdown body, working (FR-038) — via a new `src/crd/components/space/settings/CommunityGuidelinesEditor.tsx` + an integration hook (`useUpdateCommunityGuidelines`), superseding `045`'s stub; the FR-034(a) "apply a Community Guidelines template" and FR-034(b)/US9 "save current guidelines as a template" flows are wired into *this* editor (replacing whatever `045` stubbed). It uses the same title + references-list editor sub-components as the CG template form (§2). Reason: those flows need a working host; `045`'s isn't.
- `041`'s "account templates in the picker" requirement is satisfied by `TemplatePicker`'s **Account** source section (`useImportTemplateDialogAccountTemplatesQuery`); the legacy MUI dialog is not extended.

**Rationale**: prevents two specs implementing the same picker/section-list and guarantees the two holders share one experience.

**Alternatives considered**: `098` provides only primitives and `045` keeps its own tab implementation (rejected by the user — `098` owns and supersedes); `098` re-implements the entire Space Settings page (rejected — out of scope; `045` keeps the other tabs).

## 9. Async / large-content rendering — *lazy queries + Suspense + scroll areas; never block paint*

**Decision**: Template *content* (the heavy payloads — whiteboard JSON, deep space trees, callout framing) is fetched only when a preview/edit is opened, via `useTemplateContentLazyQuery`; the preview dialog and the picker preview pane render a skeleton (`role="status"` + `aria-label`) until it resolves, and put the rendered content in a `ScrollArea` so it can exceed the dialog. The section-list view shows per-card skeletons while `AllTemplatesInTemplatesSet` loads, and a per-row spinner during a delete/duplicate (already the pattern in `045`'s `useTemplatesTabData`). Mutations are wrapped in `useTransition` in the integration layer (Constitution II).

**Rationale**: FR-025 requires non-blocking preview loads; lazy-on-open is also the legacy behaviour and avoids over-fetching the list.

**Alternatives considered**: eager-fetching all template content with the list (rejected — huge over-fetch, especially in the Library where every platform template would pull its full content).
