# Feature Specification: CRD Templates System

**Feature Branch**: `098-crd-templates`
**Created**: 2026-05-11
**Status**: Draft
**Input**: User description: "Implement TEMPLATES — everything that has to do with templates: CRUD of templates; using templates when creating callouts, spaces, whiteboards, community guidelines (and more); creating and updating Innovation Packs and the templates inside spaces (the two 'template holders' — Spaces and Innovation Packs — behave almost the same today, with small differences we want to remove in the new UI); all the templates dialogs. Analyse how it's implemented in MUI and re-implement the Templates system in the CRD design system (shadcn/ui + Tailwind)."

## Context & Background

Templates are reusable, pre-configured definitions that let a creator start a new piece of content from a known-good starting point instead of from scratch. Alkemio has **five template types**:

| Type | What it pre-configures |
|---|---|
| **Callout** ("Collaboration tool" template) | A callout's framing (its kind — text / whiteboard / memo / **Collabora-document** / link / media-gallery / **poll** — plus the framing content), its contribution settings (which contribution types are allowed, comment settings), and the default content for new contributions (default post description, default whiteboard drawing). |
| **Whiteboard** | A whiteboard drawing plus its preview settings. |
| **Post** | A default post description (rich text). |
| **Space** ("Space content" template) | A space's structure: innovation-flow phases, starter callouts, nested subspace templates, and collaboration settings. Can be authored by copying an existing space. |
| **Community Guidelines** | A community-guidelines **title**, the guidelines text, plus a set of reference links. |

Every template lives in exactly one **templates set**, which is owned by a **template holder**. There are two kinds of holder today, and they behave almost identically:

1. **A Space** — owns a templates set via its *templates manager* (the templates manager also stores per-type *template defaults*: the template auto-suggested when, e.g., a member creates a subspace or a new post in a phase). Managed from **Space Settings → Templates**.
2. **An Innovation Pack** — a packaged, shareable holder with its own profile (name, description, avatar, tags, references), a *provider* organisation, a store-listing flag and a search-visibility setting. Managed from the **Innovation Pack admin** screen. Innovation Packs exist both at the **platform level** (curated, in `/admin/innovation-packs`) and at the **account level** (a user's account can own its own packs).

A third surface, the **Innovation Library** (`/innovation-library`), is a public catalogue — it aggregates all store-listed Innovation Packs and the union of their templates into a browsable, filterable gallery. It is *not* a holder (read-only).

Templates are **consumed** in many places: the callout-creation dialog ("use a template"), the whiteboard-creation flow, the space/subspace-creation flow (and the per-space "default subspace template"), the community-guidelines editor, the innovation-flow state settings ("default callout template for this state/phase"), and the callout contribution-defaults settings ("default post template"). Every one of these uses the same underlying **template picker** dialog, which shows candidate templates drawn from up to three sources: the current Space's templates set, the current account's Innovation Packs, and the public platform library.

Templates can also be **created from an existing entity** — "save as a template": a **callout** (from its settings menu), the **current community guidelines** (from the guidelines editor), or a **subspace** (from the Subspaces settings tab). Each of these opens the matching **Create Template** dialog with every field already filled in from that entity's content, fully editable before the template is saved into the chosen templates set. "Save as a template" is therefore a variant of *create*, not a separate minimal dialog.

The legacy implementation is built on Material UI. This feature **re-implements the whole Templates system on the CRD design system** (shadcn/ui + Tailwind), behind the existing CRD feature toggle (`alkemio-crd-enabled`, default OFF). The legacy MUI templates surfaces remain the default and are untouched; the CRD surfaces render only when the toggle is on. This spec **owns and supersedes** the partial CRD templates work that already exists under `src/crd/components/space/settings/` and `src/main/crdPages/topLevelPages/spaceSettings/templates/`, and the account-templates-in-picker behaviour described in spec `041-account-templates-dialog`.

### Differences between the two holders that this feature removes

In the legacy UI the two holders differ in two visible ways: (1) clicking a template in an Innovation Pack jumps straight into Edit mode (the legacy `TemplatesAdmin` `alwaysEditTemplate` flag), while clicking a template in a Space opens a read-only Preview first; (2) the Space templates surface offers an "import from library" action, the Innovation Pack one does not. They do **not** differ in which template types can be created/edited/deleted: in the legacy app *both* surfaces hard-code "all five types" — there is no per-type authorisation on a templates-management surface, the only gate is page-level access (being allowed to reach Space Settings → Templates, or `<pack>/settings`); a read-only context such as a pack's public profile offers none of the actions. The CRD re-implementation presents **one identical management surface for both holders** — same card layout, same actions (Preview, Create, Import from library, Edit, Duplicate, Delete), same dialogs — with **Preview-before-Edit everywhere**, **Import offered for a Space holder only**, and the per-type action predicates uniformly enabled in any management context and uniformly disabled in a read-only context, exactly as the legacy app behaves (i.e. no new per-type privilege check is introduced; the existing page-access gate is the gate). Holder kind never gates any action except Import.

### Session 2026-05-11

- Q: This spec covers the core Templates system in CRD (reusable template components, the create/edit/duplicate/delete/import dialogs, the Space-Settings → Templates tab, and picking templates in creation flows). Which template-adjacent areas should also be in scope? → A: **All of them** — Innovation Pack admin (create/edit a pack + manage its templates), the Innovation Library full page (`/innovation-library`), and the Innovation Pack public profile page.
- Q: How does this spec relate to the already-partially-built CRD Space-Settings → Templates tab (spec `045-crd-space-settings`) and the account-templates dialog work (spec `041`)? → A: **This spec owns and supersedes them.** The existing CRD templates code is a starting point this spec absorbs, refines and extends; spec `045` keeps the other Space-Settings tabs but the Templates tab and the template picker become this spec's responsibility.
- Q: When the two template holders are unified, which direction does "remove the difference" go for the **Import from library** action (the legacy Innovation Pack admin has no Import; Spaces do)? → A: **Import stays Space-only.** Both holders share the identical management surface, cards and dialogs, but the Import affordance is contextually present only for a Space (an Innovation Pack is a *source* of templates, not a *sink*) — handled the same way as any other authorisation-gated action. Templates still flow *into* a pack only by being created or duplicated there.
- Q: What is the import-into-Space interaction model for the template picker? → A: **Persistent library manager** — a deliberate consolidation of the two legacy import surfaces (the Space-templates import dialog, which closes after each import, and the innovation-flow default-template selector, which shows the current selection at the top with a Remove affordance): the picker shows a gallery; clicking a template previews it; "Import" adds it to the Space's set and the picker stays open; templates already in the set are marked and can be removed from there; the admin closes the picker when finished. In this mode the picker shows the **Account** and **Platform** source sections only — the Space's own templates set is not a source for importing into itself. (Consumption flows — picking a template to apply when creating a callout / whiteboard / space — remain single-pick-then-apply.)
- Q: When an admin deletes a template that is currently set as a default (default subspace/content template, or a phase's default callout template), what happens to the default reference? → A: **Auto-clear it.** Deletion proceeds; every default pointer that referenced the deleted template is cleared; the confirmation dialog notes that the template is currently used as a default and will be unset.
- Q: How is the work split with spec `045-crd-space-settings` for the in-Space-Settings template affordances (Layout-tab "default callout template per phase"; Subspaces-tab "default subspace template" + "save subspace as template")? → A: **`098` owns the Templates tab and all reusable template machinery (picker, cards, preview, create/edit forms, defaults selector); `045`'s Layout and Subspaces tabs host the per-phase-default and save-as-template / default-subspace affordances and consume `098`'s components.** Those affordances are still *specified* here (FR-032, FR-035) but their host-tab wiring is `045`'s task.
- Q: Should the spec include the "save a callout as a template" and "save the current community guidelines as a template" flows? → A: **Yes — both are in scope.** "Save a callout as a template" opens the Create *Callout* template dialog with all of the callout's details (framing kind + content, settings, contribution defaults, name/description/tags) pre-filled and editable; "save the current community guidelines as a template" opens the Create *Community Guidelines* template dialog pre-filled from the current guidelines. They join "save a subspace as a Space template" as the three "save as a template" flows. (See FR-032, US9.)
- Q: Should the spec cover **platform-level template defaults** (the `PLATFORM_*` template-default kinds)? → A: **No — out of scope.** There is no MUI UI for platform-level template defaults to migrate; this feature handles only the Space-scoped default (the default Space/content template used when creating a subspace) and the innovation-flow-state default callout template.
- Q: Do callout templates support the **Collabora-document framing kind**? → A: **Yes.** A Callout template's framing kind may be any of text / whiteboard / memo / Collabora-document / link / media-gallery; the create/edit form lets the user choose it and the preview renders it consistently with the other kinds. (If the current template-content data does not yet expose the Collabora field, that is added in-PR — a plan-level concern.)
- Q: Within a holder's templates set, is an **imported (or duplicated) template** editable, or is it a read-only "platform/imported" item? → A: **Fully editable.** An imported or duplicated template is a complete, independently-owned copy in the destination set; there is no read-only "imported" or "platform-origin" template state within a set — within any one set, a template's editability/deletability is determined by authorisation alone, not by how it got there. (This corrects the early CRD Space-Settings cut, which gated Edit/Delete to "custom only".)
- Q: The legacy Layout tab labels the per-flow-state default as "Default post template" although it is and always was a *Callout* template — which name does CRD use? → A: **"Default callout template"** (for a given innovation-flow state/phase). The legacy mislabel is corrected.

### Session 2026-05-12 (alignment with the legacy MUI implementation)

- Q: The legacy templates-management surfaces (Space Settings → Templates, the pack admin) gate template actions how? → A: **By page access only.** Both `SpaceAdminTemplatesPage` and `AdminInnovationPackPage` hard-code "all five types creatable/editable/deletable" (the Space additionally allows Import); there is no per-type privilege check. ### Session 2026-05-18 (Callout-template contribution-settings fixes)

- Q (D14): A Callout template's *response-type* chip strip (`LINKS`/`POSTS`/`MEMOS`/`WHITEBOARDS`) — what drives which chip is selected when the form opens on edit? It is currently AND-gated on `settings.contribution.enabled`, so when an admin turns **both** the "Allow Members" and "Allow Admins" switches off (`canAddContributions === NONE` ⇒ `enabled === false`), the chip resets to nothing-selected even though the template's `allowedTypes` still names the type. Is that intended? → A: **No — it is a bug.** The response-type chip selection MUST depend on `settings.contribution.allowedTypes[0]` **alone**. `settings.contribution.enabled` and `settings.contribution.canAddContributions` are an *orthogonal* concern — they control *who* (None / Admins / Members) is currently allowed to add contributions, **not** *what type* of contribution the callout collects. The two surfaces are independent: a callout can still be "a links callout" (chip-strip says LINKS) with `canAddContributions === NONE` (the two actor switches are off — nobody can add right now). This mirrors the live-callout edit mapper (`mapCalloutDetailsToFormValues.ts`), which already enforces this rule (and carries a comment explaining why AND-ing the two would round-trip-lose the response type on every edit). FR-020 amended to make the rule explicit; the template mapper is corrected to drop the `enabled &&` gate.
- Q (D15): The `UpdateCalloutTemplate` mutation's return shape currently fetches `settings.contribution { enabled allowedTypes }` only — `canAddContributions`, `commentsEnabled`, and `settings.framing.commentsEnabled` are not selected, so the Apollo cache write following an edit doesn't refresh those fields (the form keeps working only because the prior read populated them from the `CalloutTemplateContent` fragment, which uses `CalloutSettingsFull`). Should the mutation's response cover the full settings shape? → A: **Yes — extend it.** The `UpdateCalloutTemplate` mutation MUST return the full `CalloutSettingsFull` fragment on `settings`, matching what `CalloutTemplateContent` reads. This guarantees the post-update cache is consistent with the read shape (no fields silently lagging behind on a re-render) and mirrors what the live callout mutation `UpdateCalloutContent` does via `...CalloutDetails` (which itself spreads `CalloutSettingsFull`). Recorded as a confirmed fragment gap in the same V1/V2/V5 codegen-in-PR workflow.
- Q (D16): When a Callout template with **whiteboard framing** is loaded for edit / preview, the server-rendered whiteboard preview image (`framing.whiteboard.profile.preview` — the `WHITEBOARD_PREVIEW` Visual the backend stamps when a whiteboard's content changes) is currently ignored by both surfaces: the **form** shows the placeholder Excalidraw icon as if the whiteboard had never been edited, and the **template Preview dialog** renders the literal word "Whiteboard". Should both surfaces display the server preview image? → A: **Yes — both.** (a) In the **Callout template form**, when `framingKind === 'whiteboard'` and no fresh in-form blob preview exists (the user hasn't re-opened the inline whiteboard editor yet), the inline framing preview MUST display the server-rendered preview image (`framing.whiteboard.profile.preview.uri`). When the user opens the inline editor and saves, the freshly-generated blob URL takes precedence (it reflects the just-saved state); the server URL is the read-time fallback, not a competitor. (b) In the **template Preview dialog** (and any read-only preview surface — pack profile, library, picker preview pane), the whiteboard-framing branch MUST render an `<img>` of the server preview image when present, falling back to the existing placeholder text when the visual is missing. (c) Same rule extends to the **Whiteboard template** type — `WhiteboardTemplatePreview` already does this. (d) The same field flows through any callout creation flow that pre-fills from a Callout template (US3, FR-030) — the resulting callout-form's inline framing preview must show the template's server image until the user edits it. **Data plumbing.** The existing `CalloutTemplateContent` fragment already fetches `framing.whiteboard { ...WhiteboardDetails }` (and `WhiteboardDetails.profile.preview` is the Visual selected on `WHITEBOARD_PREVIEW`) — so this is **not** a GraphQL change. It is a mapper + UI-prop change: surface the URL on the `TemplateContent` callout branch and on `CalloutFormValues`, and read it where the form / preview render the whiteboard framing. FR-020 and FR-024 amended accordingly.
- Q (D17): After the user edits the inline whiteboard in a Callout template form and saves, what persists the **rendered preview image blob** so subsequent reads of the template show the new image? The user's in-form `whiteboardPreviewImages` blobs are generated by `CrdSingleUserWhiteboardDialog` but the current Callout-template create / edit paths in `useTemplateForms` send the mutation and stop — no `uploadVisual` is fired against the whiteboard's preview Visual. → A: **Mirror the live-callout connector's post-save upload step (`CalloutFormConnector` lines 305-317 for create, 447-452 for edit).** Both `CreateTemplate` (when `type: CALLOUT` with whiteboard framing) and `UpdateCalloutTemplate` already return `callout.framing.whiteboard.profile.previewVisual.id` (alias of `visual(type: WHITEBOARD_PREVIEW)`) — the integration layer reads that id and calls `uploadVisual` via `useUploadWhiteboardVisuals(previewImages, { previewVisualId })`. No mutation change is required; the response shape is already adequate. Without this step the server stores updated whiteboard content but the next `WHITEBOARD_PREVIEW` Visual fetch returns the **previous** image (or none, for a fresh template), so the form/preview keep displaying stale content. This is the third leg of D16: (a) surface the URL on read, (b) render it in the form + preview, **(c) upload the new blob on save so the server-rendered URL actually reflects the latest edit.** The same step is added to (and is already done by) the *live* callout create / edit flow — the template flow had simply not picked it up.
- Q (D20, 2026-05-19): In the Callout template's *Posts defaults* dialog (`ResponseDefaultsDialog`, opened from the "Set defaults" affordance when the response type is `post`), the user clicks "use a template", the picker loads, the user picks a Post template — but the dialog's `defaultDescription` field doesn't populate. Why? → A: **The dialog holds a local `draft` state that's only seeded on the closed → open transition.** The connector's apply path writes via parent `onSave({...values, postDescription: x})`; the parent receives the value, but the dialog's `draft` is never re-synced (only `whiteboardContent` has a sync effect — `ResponseDefaultsDialog.tsx:80-88`). The input field keeps showing the stale draft, and on Save the stale draft commits back, **overwriting** the parent's template-applied value. There's also a related Cancel bug: cancelling after a template apply silently leaves the parent with the template value even though the user discarded. **Fix**: change `ResponseDefaultsDialog`'s `templateSlot` from `ReactNode` to a render-prop that receives an `applyDraft: (next: Partial<ContributionDefaults>) => void` helper. The helper writes straight into the dialog's local `draft` (not the parent) — so the template-applied values appear immediately in the inputs, the user can review / further edit them, the dialog's Save commits the final draft, and Cancel correctly discards. Replaces the two-hop "connector → parent → sync effect → draft" loop with a one-hop "connector → draft" path. The whiteboard EDITOR sub-flow (`CrdSingleUserWhiteboardDialog` from `whiteboardSlot`) keeps its existing parent-commit path — its inputs aren't user-editable inside `ResponseDefaultsDialog`, so the existing sync effect for `whiteboardContent` continues to work without the same Cancel issue. **Dialog polish (same session)**: widen `DialogContent` from `sm:max-w-2xl` (672 px) to `sm:max-w-[720px]` (≈ +50 px) and apply the sticky-header + sticky-footer + scrollable-body layout from `TemplateFormDialog` / `TemplatePreviewDialog` — title and Save/Cancel buttons stay visible at any content height. FR-005 / FR-022 / FR-036 amended.
- Q (D19): The Callout template's **Preview dialog** has no References section — but the template form's References editor accepts entries, the server persists them, and the resulting in-feed callout *does* render them. Where did they go? → A: **The CRD callout preview mapper drops `framing.profile.references`.** `templateContentMapper.mapCalloutContent` populates `framingLinks` (the *cta* framing's external link) but never reads `framing.profile.references` (the *calloutReferences* the editor produces), and the `TemplateContent.callout` branch carries no `references` field. The legacy MUI preview renders `<References compact={true} references={…} />` between the description and the framing body (`TemplateContentCalloutPreview.tsx:58`); CRD must mirror that. **Fix**: add `references?: { id: string; name: string; uri: string; description?: string }[]` to the `TemplateContent.callout` branch, populate from `framing.profile.references` in `mapCalloutContent`, and render via the existing `ReferencesAndTagsStrip` component (the same one the in-feed `CalloutDetailDialog` uses, so styling is identical between preview and feed) — placed between the framing body and the *Allowed Contributions* section. Not a GraphQL change: `CalloutTemplateContent` already fetches `framing.profile.references` via `...ReferenceDetails`. FR-024 amended.
- Q (D18): When a user creates a **live callout** (not a template) and picks a callout template with whiteboard framing in the picker, the new callout's whiteboard ends up with no preview image — even though the template has one — unless the user re-opens the inline editor and saves. Why? `CalloutFormConnector`'s post-create upload step (lines 305-317) uploads only what the form's `whiteboardPreviewImages` array holds. When the user doesn't touch the inline editor, that array is empty (the prefill mapper only surfaces the server URL on `whiteboardPreviewServerUrl` for display; it doesn't seed blob data). So nothing gets uploaded against the new callout's whiteboard's `WHITEBOARD_PREVIEW` Visual. What should the system do? → A: **Seed the form's `whiteboardPreviewImages` from the template's server preview URL at prefill time.** In `TemplateImportConnector` (the seam between the picker and the live-callout form), after the template's full content arrives, fetch the `whiteboardPreviewServerUrl` as a `Blob` and stuff it into the prefilled values as a single `{ visualType: VisualType.WhiteboardPreview, imageData: <blob> }` entry. The existing post-create upload pipeline then carries the blob across without any new wiring — the new callout's whiteboard preview Visual receives the same image the template had. If the user opens the inline editor and saves, `CrdSingleUserWhiteboardDialog`'s save callback replaces the seeded blob with fresh ones — the freshness rule from D16 holds (blob > server URL > placeholder). If the fetch fails (CORS, network, transient 5xx), the seed is skipped silently — the user still sees the server URL in the form (the `whiteboardPreviewServerUrl` fallback from D16 stands) and worst case the new callout has no preview image, which is the same as the pre-fix behaviour — never worse. This closes the fourth leg of D16: on the live-callout side, "show the image, react to edits, **and carry it through on create**." FR-023 / FR-030 amended.

- Q (D21): The CRD subspace-creation template picker only shows **Platform** Space templates when the Create Subspace dialog is launched from the **Space view → Subspaces tab** or from a **Subspace page** — the holder Space's own Space templates (the **Space** source) and, at L0, the account's packs (the **Account** source) are missing — even though the **Space Settings → Subspaces** create flow shows them correctly. Why, and what's the fix? → A: **A callsite wiring gap, not a behaviour change.** `useCreateSubspace(spaceId, { accountId, templatesSetId, defaultTemplateId })` builds the picker's **Space** source from `templatesSetId` and its **Account** source from `accountId`, and pre-selects `defaultTemplateId` (FR-031). The Settings page (`CrdSpaceSettingsPage`, via `useSubspacesTabData`) passes all three; the two non-settings callsites (`CrdSpaceSubspacesPage` — the Space-view Subspaces tab; `CrdSubspacePageLayout` — the Subspace-page sidebar/dialog) called `useCreateSubspace(spaceId)` with **no options**, so the picker fell back to the always-present **Platform** source only. The fix wires every subspace-creation entry point with the same scope as Settings: the holder Space's `templatesManager.templatesSet.id` (Space source — sourced via `useSpaceTemplatesManagerQuery` at the L0 Space view, and read off the already-fetched `SubspacePage` query on a Subspace page), the account id (Account source — `space.accountId` at L0; left `undefined` on a Subspace, matching the Settings scope which itself leaves account `undefined` at L1/L2), and the configured `SPACE_SUBSPACE` default template (pre-selected per FR-031). No GraphQL, mapper, or component change — purely passing the options the hook already accepts at the two callsites that omitted them. FR-031 amended to make explicit that the Space-template selector MUST be wired with the holder's own templates set + default at **every** subspace-creation surface, not only in Settings.

- Q (D23): In the Create Subspace dialog the "Upload Avatar" / "Upload Card Banner" buttons open the OS file picker but **no file is selectable** (everything greyed out). Why? → A: **The `<input type="file">` `accept` attribute is malformed.** `CreateSubspaceDialog`'s `FileField` builds `accept` from `constraints.allowedTypes` by prepending a dot to each entry (`ext.startsWith('.') ? ext : \`.${ext}\``). But `VisualConstraints.allowedTypes` (from `platform.configuration.defaultVisualTypeConstraints`) holds **MIME types** (e.g. `image/png`, `image/jpeg`) — not bare extensions — so the mapping yields invalid `accept` tokens like `.image/png`, which the browser cannot match against any file, filtering out everything. The rule: a file-input `accept` built from `VisualConstraints.allowedTypes` MUST use the MIME types **verbatim** (`allowedTypes.join(',')`), matching the legacy `FileUploadWrapper` (`accept={allowedTypes.join(',')}`) and `UploadButton` (documented as "Array of MIME types"). Falling back to `image/*` only when the constraint list is empty. Pure CRD-layer fix; no prop or behaviour change.

- Q (D22): The Create Subspace dialog (`CreateSubspaceDialog`) scrolls as a single block — its title and the Cancel/Create footer buttons scroll out of view on tall content (template preview + markdown body + visuals). Should it adopt the sticky-header / sticky-footer / scrollable-body layout the other CRD template dialogs use? → A: **Yes.** `CreateSubspaceDialog` MUST use the same shell layout as `TemplateFormDialog` / `TemplatePreviewDialog` / `ResponseDefaultsDialog` (D20): `DialogContent` is `flex flex-col max-h-[90vh] p-0 gap-0 overflow-hidden`; the `DialogHeader` is a fixed top band (`border-b`, padded) that always shows the title/subtitle; only the body (`flex-1 overflow-y-auto`) scrolls; the `DialogFooter` is a fixed bottom band (`border-t`, padded) that always shows Cancel/Create. Pure CRD-layer styling change — no prop or behaviour change. Generalises the established template-dialog shell convention to the subspace-creation dialog.

- Q (D24): The Innovation Pack edit form's **References** section (`InnovationPackForm`) used the `crd-templates` `ReferenceRowsEditor`, which has no way to **attach a file** — unlike the legacy references field and the CRD Create-Callout dialog. Should it gain a paperclip file-upload, sharing one component with the profile forms (spec 097) and the Space About references (spec 045)? → A: **Yes.** `InnovationPackForm` MUST render the shared `@/crd/forms/references/ReferencesEditor` (canonical row `{ id?, name, uri, description? }` — the same shape `ReferenceRowsEditor` used, so a drop-in for the rows) instead of `ReferenceRowsEditor`. It gains `onReferenceFileUpload` / `referenceUploadAccept` props; the connector `useInnovationPackAdmin` wires them via `useStorageConfig({ locationType: 'innovationPack', innovationPackId })` → `useReferenceFileUpload(storageConfig)` so uploads land in the pack's own storage bucket (`temporaryLocation: true`). The existing save fan-out (`updateInnovationPack` for existing refs + `createReferenceOnProfile` for new + `deleteReference` for removed — keyed on rows without an `id`) is unchanged. Pure presentation + integration wiring; no GraphQL change. **The same applies to the *live* community-guidelines editor (FR-038, Space Settings → Community):** it renders inside the space's ambient `StorageConfigContextProvider`, so it MUST likewise render the shared `@/crd/forms/references/ReferencesEditor` (replacing `ReferenceRowsEditor`) and gain `onReferenceFileUpload` / `referenceUploadAccept` props; the integration hook `useCommunityGuidelinesData` wires them via `useReferenceFileUpload(useStorageConfigContext())` so uploads land in the space's own bucket (`temporaryLocation: true`). Its save fan-out (`createReferenceOnProfile` for new rows + the bulk `updateCommunityGuidelines` for existing rows) is unchanged. (Community-guidelines **template** forms also render the shared `@/crd/forms/references/ReferencesEditor`, **with** the paperclip. The earlier assumption that template forms have no storage bucket at edit time was wrong: both template-host surfaces wrap their subtree in a `StorageConfigContextProvider` — the Space Templates tab in the space bucket (ambient `useStorageConfigContext()`), the Innovation Pack admin in the pack bucket (`useStorageConfig({ locationType:'innovationPack', innovationPackId })`) — and `useReferenceFileUpload` uploads with `temporaryLocation: true`, so a file attached **before** the template entity exists lands in the holder's bucket temporarily and is claimed when the reference is saved. `useTemplateForms` accepts a `referenceUpload` arg (`{ onFileUpload?, accept? }`, mirroring its `markdownUpload` wiring) that the two provider-wrapped callsites pass and read-only / test callers omit; it forwards `onReferenceFileUpload` / `referenceUploadAccept` to the CG template form for **both** create and edit (the form is the same in either intent). With this, the bespoke `crd-templates` `ReferenceRowsEditor` is **removed** — the name/uri references list is now the single shared `@/crd/forms/references/ReferencesEditor` everywhere, paperclip included. The **Callout** template form (`CalloutTemplateForm`, integration layer) is the one component-level exception: it keeps the callout-layer `@/crd/forms/callout/ReferencesEditor` (title/url shape) it shares with the live Create-Callout dialog, which already renders the paperclip via the shared `RowAttachFileButton` — so its fix is purely wiring the same `useTemplateForms` `referenceUpload` through to it (holder bucket, `temporaryLocation: true`), giving the paperclip in callout-template create and edit just like the live dialog. The save-callout-as-template flow opened from `CalloutSettingsConnector` passes neither `markdownUpload` nor `referenceUpload`, matching its existing wiring — that template form has no image/file upload, unchanged here.)

## User Scenarios & Testing *(mandatory)*

> All user stories assume the CRD feature toggle is ON. When it is OFF the corresponding legacy MUI surface renders unchanged.
>
> Priority labels order the work for incremental delivery; every story is independently testable. The MVP is **US1 + US3** (a space admin can manage their templates, and a creator can use a callout template) — that alone is a coherent, shippable slice.

### User Story 1 — Manage a Space's templates (Priority: P1)

A Space admin opens **Space Settings → Templates** and sees the space's templates organised by type — one section per type (Space, Collaboration tools, Whiteboard, Post, Community Guidelines), each section showing the templates as cards with a name, description, type, and (where present) a banner/preview image. From here the admin can: **preview** any template (a dialog showing the template's content rendered read-only); **create** a new template of any of the five types; **edit** a template's details (and, for the editable parts of its content, the content); **duplicate** a template; **delete** a template (always behind a confirmation dialog); and **import** one or more templates into the space's set from the library (account packs and the public platform library). The admin can also choose the space's **default Space (content) template** — the one pre-selected when someone creates a subspace. (Once an admin can reach this surface they may take every one of these actions on every template type — the gate is page access, not a per-type privilege; see FR-014.)

**Why this priority**: This is the central, highest-traffic management surface and the foundation everything else builds on (the cards, the preview dialog, the create/edit forms, the import/library dialog are all reused by the other stories). It also unblocks the in-progress CRD Space Settings work.

**Independent Test**: Enable CRD, open a space's Settings → Templates as an admin. Create a template of each type, edit it, duplicate it, preview it, delete it (confirming the dialog), and import a template from the library. Set the default subspace template. Verify each action persists and the section list updates.

**Acceptance Scenarios**:

1. **Given** a Space admin opens Settings → Templates with CRD enabled, **When** the page renders, **Then** the templates appear grouped into one section per type, each showing template cards (and a count per section), and a Create / Import affordance is visible for every type.
2. **Given** the admin clicks a template card, **When** the preview opens, **Then** the template's content renders read-only in a CRD dialog (callout framing, whiteboard drawing, post description, space structure, or guidelines text + references — appropriate to the type), with an "Edit" action shown if the admin may edit it.
3. **Given** the admin chooses "Create new" for a type, **When** the create dialog opens and the admin fills the type-appropriate fields and submits, **Then** a new template of that type is added to the space's set and appears in its section.
4. **Given** the admin chooses "Edit" on a template, **When** they change its details (name, description, tags, banner) and any editable content and save, **Then** the changes persist and the card updates.
5. **Given** the admin chooses "Duplicate" on a template, **When** they confirm, **Then** a copy of the template is added to the space's set.
6. **Given** the admin chooses "Delete" on a template, **When** the CRD confirmation dialog appears and they confirm, **Then** the template is removed from the set; **When** they cancel, **Then** nothing changes.
7. **Given** the admin opens "Import from library", **When** the picker shows templates from the account's packs and the public platform library, **Then** the admin can preview a candidate and import it; the imported template appears in the space's set.
8. **Given** the admin opens the "default subspace template" selector, **When** they pick one of the space's Space templates (with a preview of its phases/callouts), **Then** that template becomes the space's default for subspace creation.
9. **Given** a Space admin who can reach Space Settings → Templates, **When** the page renders, **Then** Create / Edit / Duplicate / Delete are offered for every one of the five template types (and Preview always), and Import is offered for every type — the surface gates at **page access, not per template type** (FR-014), so there is no "this type is read-only for you" state here; and every type's section is shown even when it has no templates (with its Create / Import affordances). (A type's section is hidden when empty *only* in a read-only context — e.g. a pack's public profile — see FR-015 / US8.)

### User Story 2 — Manage an Innovation Pack's templates (unified holder experience) (Priority: P1)

A user with rights over an Innovation Pack (a platform admin in `/admin/innovation-packs`, or an account admin for an account-owned pack) opens the pack's admin screen and manages the pack's templates using the **same surface, same cards, same actions and same dialogs** as US1 — Preview, Create, Edit, Duplicate, Delete — gated only by **page access** (the same way the Space Settings → Templates tab is gated), not per template type. The legacy holder-specific quirks are gone: clicking a template opens a Preview (not an immediate Edit). The one intentional contextual difference from a Space holder is that **Import from library is not offered for a pack** (a pack is a *source* of templates, not a sink — templates enter a pack only via Create or Duplicate). Alongside the templates, the screen shows the pack's own details for editing (covered by US7).

**Why this priority**: Innovation Packs are the *other* template holder and the user explicitly wants the two unified. Treating this as P1 forces the management surface to be holder-agnostic from the start rather than re-fitted later.

**Independent Test**: Enable CRD, open an Innovation Pack's admin screen. Verify the templates area looks and behaves exactly like the Space Settings → Templates tab: create / edit / duplicate / delete / preview all work the same way and through the same dialogs. Confirm that clicking a template opens a preview first (not Edit), matching the Space behaviour, and that the only contextual difference is the absence of the Space-only "Import from library" affordance.

**Acceptance Scenarios**:

1. **Given** a user with pack-management rights opens an Innovation Pack admin screen with CRD enabled, **When** it renders, **Then** the templates area is visually and behaviourally identical to the Space Settings → Templates tab (same section-per-type layout, same card design, same action set), except the "Import from library" affordance is not shown (Import is Space-only).
2. **Given** the user clicks a template card in a pack, **When** it opens, **Then** a read-only Preview dialog appears first — *not* an immediate Edit form — exactly as in a Space.
3. **Given** a user who can reach the pack admin screen, **When** it renders, **Then** Create / Edit / Duplicate / Delete are offered for every one of the five template types (and Preview always) — identical to the Space Settings → Templates tab — because both surfaces gate at **page access, not per template type** (FR-014); the gating never depends on the holder being a pack. (A user without that access doesn't reach the admin screen at all; on the pack's *public profile* they get Preview only — US8.)
4. **Given** templates can only enter a pack by being created or duplicated there, **When** the user adds a template to a pack, **Then** they do so via the same Create / Duplicate dialogs used in a Space; there is no "import into pack" path.
5. **Given** any create / edit / preview / delete / duplicate action is taken in either a Space or a pack, **Then** it goes through the identical CRD dialog component — there is no separate "pack version" of any template dialog.

### User Story 5 — Use a template in the remaining creation flows (Priority: P2)

The CRD template picker is wired into the remaining template-consumption points so that, with CRD enabled, every place the legacy app lets a user apply a template has a consistent CRD equivalent:

3. **Given** the source entity has unsaved changes, **When** the user chooses "Save as a template", **Then** they are prompted to save or discard those changes first (mirroring the legacy behaviour), and the template is captured from the saved state.
4. **Given** a template was created via "Save as a template", **When** the user later opens it from the templates-management surface, **Then** it is editable / duplicable / deletable on the same terms as any other template in that set (it is not a read-only item).
5. **Given** the legacy source surface would not offer "save as a template" to this user (its existing gate — e.g. the callout-settings "save as template" permission, the Subspaces-tab privileges), **When** they look at the source entity in CRD, **Then** the "Save as a template" affordance is likewise not offered.

### Edge Cases

- **Empty type sections**: a holder with no templates of a given type shows that type's section with an empty state and a Create / Import affordance if the user may add to it; if the user may not add to it and there are no templates, the section is hidden entirely.
- **Empty picker sources**: in the template picker, a source section (Space / Account / Platform) with no matching templates is omitted; if *all* sources are empty, the picker shows a "no templates available" message rather than an empty dialog. The **Space** source is also omitted (regardless of contents) when the picker is importing into that same Space's set — a set is not a source for importing into itself (this matches the legacy import-into-Space dialog, which disables the Space templates section).
- **Deep-linking a template**: the legacy templates surfaces accept a URL that targets a single template (`<templates-base-url>/<templateId>` on the Space tab; `<pack.profile.url>/settings/<templateId>` in the pack admin). The CRD surfaces MUST preserve these URLs (FR-007) — a CRD deep-link resolves to that template's **read-only Preview dialog** (with an Edit affordance when the user may edit it), on *both* the Space tab and the pack admin, consistent with FR-013's "Preview-before-Edit everywhere" (the legacy "pack deep-link opens straight into Edit" behaviour is intentionally dropped — see Clarifications, D1).
- **Picking a Space template over unsaved data**: changing the selected Space template while the create-space / create-subspace form holds user-entered data requires a confirmation before the data is replaced.
- **Deleting a template that is in use as a default**: deleting a template currently set as the default subspace/content template, or as a phase's default callout template, must not leave a dangling default — the deletion proceeds and every default pointer referencing that template is cleared as part of the deletion; the confirmation dialog notes that the template is currently used as a default and will be unset.
- **Templates with uploaded images**: callout-template media-gallery images and whiteboard-template preview images load lazily in cards and previews; when an image is missing, the card uses the design system's deterministic colour fallback (per the CRD migration guide) — never a stock placeholder.
- **Editing nested Space-template content**: editing a Space template from the management surface edits its details (name, description, tags, banner); the template's nested structure (phases, callouts, subspace templates) is captured at authoring time (by copying a space or a subspace) and is not deep-edited field-by-field in this surface — re-capturing from an updated space is the path to changing it. Whether deeper in-place editing is offered is out of scope here.
- **Name collisions on import / duplicate**: importing or duplicating a template whose name matches an existing one in the destination is allowed (template names are not unique within a set); the new template is added as-is, mirroring legacy behaviour.
- **Callout template with Collabora-document or poll framing**: the template stores the framing kind ("Collabora document" / "poll") and its starting content/metadata; the create/edit form lets the user choose either kind like the others. The preview renders a Collabora-document framing's title/placeholder read-only via the existing CRD Collabora preview component (the live collaborative-document service is not embedded inside a template preview); a poll framing's question/options render read-only. If the current `CalloutTemplateContent` data does not expose the Collabora-document and/or poll framing fields (it currently does not), that gap is closed in-PR — a `.graphql` fragment edit + `pnpm codegen` + a committed schema diff (the schema enum already has both `CollaboraDocument` and `Poll`) — a plan/implementation concern, not a behaviour change.
- **No read-only templates within a set**: there is no "platform" or "imported" state that makes a template uneditable inside a holder's set. Whether a template came from Create, Duplicate, Import-from-library, or "Save as a template", once it is in a set it is editable / duplicable / deletable on the same authorisation terms as any sibling.
- **"Save as a template" from an entity with unsaved changes**: choosing "save as a template" on a callout / the current community guidelines / a subspace that has unsaved edits prompts the user to save or discard those edits first; the template is then captured from the saved state.
- **Anonymous users**: signed-out visitors can browse the Innovation Library and any pack's public profile and can preview templates; they are never shown management actions.
- **Concurrent edits**: two admins editing the same template concurrently — last write wins, mirroring legacy behaviour; no new locking is introduced.
- **Large content in preview**: very large whiteboard drawings or deep space trees render progressively / within a scroll area; the preview dialog must not block the UI while loading content.
- **Toggle off mid-session**: turning the CRD toggle off returns every templates surface to its exact legacy MUI behaviour with no data effect.
- **A holder the user can view but not manage**: a user who can see a Space's settings or a pack's profile but lacks management privileges sees templates and previews but no Create / Edit / Delete / Import.

**Per-type create / edit forms**

- **FR-020**: Creating or editing a template MUST present, in addition to the common details (name, description, tags, banner/visual), the type-appropriate content fields:
  - **Callout** template — the framing kind (text / whiteboard / memo / **Collabora-document** / link / media-gallery / **poll**) and its framing content, the allowed contribution types and comment settings, and the contribution defaults (default post description, default whiteboard drawing). The create/edit form for a Callout template MUST cover **every** framing kind the live callout editor supports (including the **Collabora-document** and **poll** kinds — see FR-007 and the GraphQL note in the edge cases). **Tag-set distinction (callout-only).** Because the *captured callout* itself carries tags, a Callout template has **two** editable tag fields, both persisted: the template-level tags (the common field shared with every type — rendered by the dialog shell, drives template-gallery filtering) **and** the captured callout's own tags (rendered inside the Callout-template body slot — applied to every callout created from this template). The mapper that prefills the form from a loaded template (`calloutTemplateContentToFormValues`) MUST mirror the live-callout edit mapper's rule (read the chip from `allowedTypes` alone), otherwise the chip silently resets to "none" on every edit-dialog open whenever the toggles are off and the update mapper would commit that loss back to the server.

A failed fetch is non-fatal (CORS, network, 5xx) — the seed is silently skipped; the form still displays the server URL via the D16 fallback and worst-case the new callout has no preview image (the pre-fix behaviour, never worse).

### Key Entities

- **Template** — a reusable, named definition with a profile (display name, description, tags, banner/visual) and a `type` (Callout, Whiteboard, Post, Space, or Community Guidelines). Belongs to exactly one **templates set**. Its *content* depends on its type (see Context).
- **Templates set** — the collection of templates owned by a holder, partitioned by type.
- **Template holder** — anything that owns a templates set. Two kinds: a **Space** (via its **templates manager**) and an **Innovation Pack**. The Innovation Library is *not* a holder (read-only aggregation).
- **Templates manager** — a Space's wrapper around its templates set plus its per-type **template defaults**.
- **Template default** — a per-type pointer on a Space marking "use this template by default" (e.g. default subspace template; default callout template per innovation-flow phase).
- **Innovation Pack** — a shareable holder of templates with its own profile (name, description, avatar, tags, references), a **provider** organisation, a store-listing flag and a search-visibility setting. Exists at platform level and at account level.
- **Innovation Library** — the platform catalogue: the set of store-listed Innovation Packs plus the union of their templates, browsable and filterable by type.
- **Template picker** — the dialog used wherever a user selects a template to apply (callout / whiteboard / space / subspace / community-guidelines / phase-default / post-default / import-into-holder); shows candidates grouped by source (current Space's set, current account's packs, public platform library) and supports preview before selection.
- **Account** — the owner of account-level Innovation Packs and the source of "account templates" in the picker (the union of the account's packs' templates).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: With CRD enabled, a Space admin can complete the full lifecycle — create, edit, duplicate, preview, delete, and import — for **every one of the five template types** entirely within the CRD experience, with no MUI fallback and no missing action, in a single session.
- **SC-002**: With CRD enabled, **every** template-picking-or-creating interaction available in the legacy MUI app — callout creation from a template, whiteboard creation from a template, space/subspace creation from a template, default subspace template, community-guidelines application, innovation-flow-state default callout template, post default description, import-into-holder, and "save as a template" for a callout / the current community guidelines / a subspace — has a working CRD equivalent, verified against a checklist with zero gaps (allowing only the documented "host surface not yet CRD" transitional fallbacks, each explicitly listed).
- **SC-003**: A side-by-side comparison of the templates-management surface for a Space and for an Innovation Pack shows an **identical** layout, card design, action set, and dialog set — a reviewer cannot tell which holder kind they are looking at from the templates area alone.
- **SC-004**: Across all CRD templates surfaces, **no Material UI styling layer is loaded** (verifiable from the loaded bundles for those routes/dialogs).
- **SC-005**: A signed-out visitor can open `/innovation-library` and any Innovation Pack's public profile with CRD enabled, browse and filter templates, and preview any template — with no authentication prompt and no errored region.
- **SC-006**: Toggling the CRD switch OFF restores the exact legacy MUI templates experience on every surface, with no residual data or state effect.
- **SC-007**: Every CRD templates surface passes an automated accessibility check (WCAG 2.1 AA) and a keyboard-only walkthrough of its primary task (e.g. create a template, pick a template, delete a template) with no blocking issue.
- **SC-008**: Every destructive action on a CRD templates surface (template delete, pack delete, guidelines replacement, discard unsaved form) is preceded by a confirmation dialog — verified by exercising each path.

## Assumptions

- **Toggle & coexistence**: This work ships behind `alkemio-crd-enabled` (default OFF); the legacy MUI templates surfaces remain the default and are untouched, consistent with the rest of the CRD migration. Removal of the toggle and deletion of legacy files happens later, project-wide, not in this feature.
- **Data layer unchanged (with a small set of additive fragment edits)**: The GraphQL schema, queries, mutations and data hooks for templates, templates sets, templates managers, template defaults, Innovation Packs and the platform library are reused as-is; this feature changes only the presentation layer and its integration glue. The exceptions are additive `.graphql` *fragment* edits + `pnpm codegen` + committed schema diffs for: **V1** Collabora-document framing on `CalloutTemplateContent`; **V2** poll framing on `CalloutTemplateContent` (FR-020/FR-030); **V3** the data needed for FR-019's delete-cascade (which defaults / flow-states reference a deleted template — only if the backend doesn't cascade and the client must learn it); **V5** the `title` field on `CommunityGuidelinesTemplateContent` (FR-020/FR-038); **V6** the `UpdateCalloutTemplate` mutation's `settings.*` return shape is widened to the full `CalloutSettingsFull` fragment so the cache-write following an edit covers `canAddContributions`, `commentsEnabled` and `framing.commentsEnabled` (not just `enabled` + `allowedTypes`) — matching what `CalloutTemplateContent` reads (D15, 2026-05-18). These look like fragment-only changes (the relevant schema fields/enums already exist) — the first Phase-2 task verifies each server-side, and anything that needs a real backend change is flagged as a blocker before the dependent UI is built. No new GraphQL *documents* are introduced.
- **Ownership of prior CRD work**: The partial CRD templates code under `src/crd/components/space/settings/` (the templates view, the library/edit/preview dialogs, the save-subspace-as-template and change-default-subspace-template dialogs) and `src/main/crdPages/topLevelPages/spaceSettings/templates/` is a starting point this feature absorbs, refines and extends; spec `045-crd-space-settings` continues to own the other Space-Settings tabs. The account-templates-in-picker behaviour from spec `041-account-templates-dialog` is folded into the CRD picker built here.
- **Reusable picker**: A single CRD template-picker dialog (with preview + free-text search) is built once and consumed by every flow (import-into-holder and all creation/default flows); it shows the Account and Platform source sections always (omitting empty ones) and the Space source section when applicable — omitted in import-into-that-same-Space mode and where there is no Space context.
- **Holder unification direction**: "Removing the small differences" means converging on the richer, Space-style behaviour: Preview-before-Edit everywhere, the same action set and dialogs everywhere; the per-type action predicates are uniformly enabled in a management context and uniformly disabled in a read-only context, matching the legacy page-access gate (there is no per-type privilege check in the legacy app). The one intentional contextual difference that remains is that Import-from-library is offered for a Space holder and not for an Innovation Pack (a pack is a template *source*); templates enter a pack only via Create or Duplicate (resolved in Clarifications, 2026-05-11).
- **Innovation Pack admin/profile placement & deletion**: The CRD Innovation Pack admin and public profile pages live at the same URLs as today — `<pack.profile.url>/settings` (admin) and `<pack.profile.url>` (public profile) — both mounted in `src/domain/InnovationPack/InnovationPackRoute.tsx` under `TopLevelRoutes.tsx` (the platform-admin pack *list* at `/admin/innovation-packs` is a separate page, not migrated by this feature). Pack **creation** is from the "Account" tab of a user's/org's profile (`accountId` in scope there); pack **deletion** is from the three-dot menu on each pack card in that same Account-tab packs list. The provider organisation is the account's host organisation, shown read-only.
- **Innovation Library scope**: The CRD Library page reproduces the legacy page's content — an Innovation Packs section and a type-filterable template gallery with preview — restyled in CRD; no new catalogue features are introduced beyond what the legacy page offers.
- **Space-template content editing depth**: Editing a Space template in the management surface edits its details (name, description, tags, banner); the nested structure is (re-)captured by copying a space/subspace, not deep-edited field-by-field here — matching the current behaviour.
- **"Save as a template" is a create variant**: "save a callout / the current community guidelines / a subspace as a template" opens the matching Create Template dialog **pre-filled** from that entity — there is no separate minimal "save as" dialog. The CRD callout-settings menu's existing "Save as template" (which today opens a legacy MUI dialog) is re-pointed to the CRD Create Callout-template dialog.
- **Callout framing kinds**: a Callout template can represent every callout framing kind the live callout editor supports, including the **Collabora-document** and **poll** kinds. The Callout template create/edit form reuses the existing CRD callout-authoring connectors (whose framing-kind type is the `FramingChip` union — `'none' | 'whiteboard' | 'memo' | 'document' | 'cta' | 'image' | 'poll'`, defined in `src/crd/forms/callout/types.ts`); the `crd-templates` `types.ts` re-exports it rather than defining a parallel union. If the existing `CalloutTemplateContent` GraphQL fragment does not expose the Collabora-document and/or poll framing fields (it does not), they are added in-PR via codegen with a committed schema diff (a plan-level concern; not a change to the templates data model itself).
- **Community-guidelines editor delivered here**: the CRD community-guidelines editor (Space Settings → Community) is delivered by this feature to legacy parity — guidelines title + reference links + markdown body, working (FR-038) — superseding `045`'s markdown-only partial version; FR-034's apply / save-as affordances are wired into it.
- **Deep-links open Preview**: a CRD deep-link to a single template (`<base>/<templateId>`, `<pack>/settings/<templateId>`) opens that template's read-only Preview dialog (Edit affordance when authorised) on both surfaces — the legacy "pack deep-link → Edit" behaviour is dropped (Clarifications D1, FR-013).
- **Single-PR delivery**: this feature ships as one PR; the `tasks.md` phases are internal build-order milestones, not separate merge points (the natural split-line if the PR grows unmanageable is Phase 1+2+US1 ∣ the rest).
- **No read-only templates within a set**: imported / duplicated / saved-as templates are full owned copies; the early CRD Space-Settings cut that gated Edit/Delete to "custom only" is corrected — within a management context every template in the set is editable/duplicable/deletable (the legacy `TemplatesAdmin` has no per-template gate), not gated by template origin.
- **i18n**: Templates strings live in a CRD `templates` feature namespace under `src/crd/i18n/templates/`, maintained manually (AI-assisted) for all supported languages in the same PR, per the CRD i18n conventions. Business-domain text (template names, descriptions) flows in as props.
- **Reused CRD building blocks**: Existing CRD primitives/components are reused — the confirmation dialog, the markdown editor, the tags input, the share dialog, the deterministic colour fallback for missing banners/avatars, date-fns for any date formatting, the CRD callout-authoring connectors (framing connectors incl. Collabora, the callout-settings controls, contribution-defaults), the CRD whiteboard editor/viewer and its preview-settings/crop dialogs, the shared whiteboard "configure / edit drawing" row component (`WhiteboardConfigCard` in `src/crd/components/whiteboard/` — the "New whiteboard · Ready to be created · Configure" card used by both the callout whiteboard-framing editor and the whiteboard-template form), the CRD Collabora preview component — rather than re-created.

## Dependencies

- **CRD design system & toggle**: `src/crd/` primitives/components, the `CrdLayoutWrapper` / route-gating mechanism, and `useCrdEnabled` (existing).
- **Spec `045-crd-space-settings`**: provides the CRD Space Settings shell (incl. the Layout and Subspaces tabs). This feature owns the **Templates tab** and all reusable template machinery; `045`'s Layout tab (per-phase default callout template) and Subspaces tab (default subspace template, save-subspace-as-template) **host** those affordances and **consume this feature's components** — those flows are specified here (FR-032, FR-035) but wired into the tabs by `045`. This feature also **delivers the CRD community-guidelines editor** (FR-038): `045`'s Community-tab editor is markdown-only and incomplete, so `098` supersedes it — the FR-034 apply / save-as affordances are wired into the editor this feature builds (replacing whatever `045` stubbed).
- **Spec `041-account-templates-dialog`**: its account-templates-in-picker requirement is satisfied by the CRD picker built here (the spec's intent carries over; its MUI implementation is not extended).
- **CRD callout / whiteboard / space-creation surfaces**: the consumption flows (US3, US4, US5) wire the CRD picker into these surfaces; for any that are not yet CRD, the legacy picker remains until they migrate (FR-037). The Callout template create/edit form (FR-020) and the "save a callout as a template" flow (US9 / FR-032) reuse the existing CRD callout-authoring components (framing editors for each kind including Collabora-document, contribution settings, contribution defaults) rather than re-creating them.
- **Existing template GraphQL hooks & mutations** for templates sets, templates, template defaults, Innovation Packs, and the platform library (`src/core/apollo/generated/apollo-hooks.ts` and the `.graphql` documents under `src/domain/templates/`, `src/domain/templates-manager/`, `src/domain/InnovationPack/`).

## Out of Scope

- Changing the templates data model, the GraphQL schema, or template authorisation rules (mirror, don't change).
- Removing the CRD feature toggle or deleting the legacy MUI templates code (a later, project-wide step).
- Migrating to CRD any *host* surface that merely *contains* a template-picking step but is not itself a templates surface (e.g. the full callout-creation dialog, the full whiteboard editor, the full top-level create-Space wizard) — those migrations are owned by their own specs; this feature only delivers the picker and wires it in where the host is already CRD. **One exception**: the **CRD community-guidelines editor** (Space Settings → Community) — `045`'s version is markdown-only and incomplete, so this feature delivers it to legacy parity (title + references + body — FR-038) as the host for FR-034's apply / save-as flows.
- New template types, new catalogue/discovery features, template versioning, template analytics, or any capability the legacy app does not already have — **with one explicit exception**: "save the current community guidelines as a template" (FR-034(b) / US9), a deliberate small addition for parity-of-experience with the callout and subspace "save as a template" flows. (Bringing the CRD community-guidelines *editor* to legacy parity — title + references + body, FR-038 — is **not** a new capability: it is the legacy MUI editor's feature set re-implemented in CRD, finishing `045`'s half-done version. Adding poll framing to Callout templates — FR-020/FR-030 — is likewise no-regression: the live callout editor already supports poll framing, templates simply could not represent it before.)
- The **platform-admin Innovation Pack list page** (`/admin/innovation-packs` → `AdminInnovationPacksPage`) — it stays the legacy MUI page (FR-037 transitional); a platform admin reaches the CRD pack admin screen by opening a pack from that list. Platform-level pack creation/deletion stays where the legacy app puts it (the list page and the account-tab pack-card menu).
- Choosing or changing a pack's **provider organisation** — the provider is the owning account's host organisation and is shown read-only; reassigning a pack's provider is done by the platform-admin "Transfer Innovation Pack" feature, which is out of scope here.
- Deep field-by-field in-place editing of a Space template's nested structure (phases/callouts) beyond what the legacy app offers.
- **Platform-level template defaults** (the `PLATFORM_SPACE`, `PLATFORM_SPACE_TUTORIALS`, `PLATFORM_SUBSPACE`, `PLATFORM_SUBSPACE_KNOWLEDGE` template-default kinds): there is no MUI UI for these to migrate, so they are not covered here. This feature handles only the Space-scoped default (the default Space/content template for subspace creation, FR-018) and the innovation-flow-state default callout template (FR-035).
