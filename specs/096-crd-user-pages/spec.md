# Feature Specification: CRD Public Profile Pages (User, Organization, Virtual Contributor)

**Feature Branch**: `096-crd-user-pages`
**Created**: 2026-04-29 (expanded 2026-04-30 to add Organization + VC profile pages)
**Status**: Draft
**Input**: Migrate the public **profile pages** for the three actor types — **User** (`/user/:userSlug`), **Organization** (`/organization/:orgSlug`), and **Virtual Contributor** (`/vc/:vcSlug`) — from the current MUI implementation to the CRD design system (shadcn/ui + Tailwind).

**Per-actor design source** (this is the canonical rule for every section / clarification below):

- **User profile** — **redesign per the prototype at `prototype/src/app/pages/UserProfilePage.tsx`**. The prototype introduces a 5-tab resource strip, drops the MUI's flat list of sections, and omits the MUI's innovation-packs / innovation-hubs sub-sections. This page is **not** a parity restyle of the current MUI; where prototype and MUI disagree, the prototype wins.
- **Organization profile** — **parity restyle of the current MUI** (`OrganizationPageView` + `AssociatesView`). No prototype exists; no design changes vs. today's MUI behavior.
- **VC profile** — **redesign per the (updated) prototype at `prototype/src/app/pages/VCProfilePage.tsx`**. The prototype rebuilds the **hero** (adds a "Virtual Contributor" badge and skill-tag chips, drops the banner image) and the **right column** (replaces today's three hard-coded blocks with a **Functionality** card grid, an **AI Engine** card grid, and a **Monitoring** strip — each driven by the existing `modelCard.spaceUsage` / `modelCard.aiEngine` / `modelCard.monitoring` data). The **sidebar** (Description / Host / References / Body of Knowledge) remains a parity restyle of the current MUI with only minor tweaks (sticky positioning; References renders all entries instead of MUI's silent split-and-discard of social refs). The **logic behind** the redesigned right column — the data extraction currently encapsulated in `useTemporaryHardCodedVCProfilePageData(modelCard)` — is reused as-is in the data mapper; only the labels, descriptions, and the `<strong>`-tagged copy that today live as hardcoded English strings (with `dangerouslySetInnerHTML`) are replaced by i18n keys in the `crd-profilePages` namespace and rendered via `<Trans>`. Where prototype and MUI disagree, the prototype wins for visual structure and the MUI wins for data sources, branching logic, and the three BoK variants. Where the prototype shows only one variant of a multi-variant block (e.g., the AlkemioSpace BoK case), the MUI's other variants (KnowledgeBase, External, no-BoK) are still ported using the existing parity rules.

Follow the migration pattern established by spec `045-crd-space-settings` and the architectural rules in `docs/crd/migration-guide.md`. The seven **User Settings** tabs (`/user/:userSlug/settings/*`) are owned by sibling spec `097-crd-user-settings`. Both specs share the same `useCrdEnabled` toggle and the shared layout shell `CrdLayoutWrapper`. Organization and VC settings/admin pages are **not** in scope here — they remain future specs.

## Clarifications

### Session 2026-04-29 (User profile)

- Q: Online-presence dot on the public User profile hero — derive from `lastActiveDate`, drop entirely, or wire a real presence channel? → A: **Drop entirely.** The dot is removed from the public profile hero. This keeps the migration a parity restyle (current MUI does not surface presence) and avoids implying real-time online status without an actual presence channel. `User.lastActiveDate` exists on the schema but stays unused by CRD in this iteration.
- Q: Does the public User profile Message button stay in the CRD hero? → A: **Yes — keep it.** The CRD hero exposes a Message button for any signed-in non-owner viewer. The button reuses the existing `useSendMessageToUsersMutation` via a callback prop, mirroring the current MUI `UserPageBanner.handleSendMessage` flow exactly: clicking the button opens an in-hero compose surface (or a small dialog), the user types a message, the mutation fires with `{ message, receiverIds: [userId] }`, and on success the surface closes. No new GraphQL mutation is introduced.
- Q: Public-profile resource sections — pagination strategy? → A: **Per-actor rule (FR-016).** **User profile** follows the prototype (`prototype/src/app/pages/UserProfilePage.tsx`): every resource section renders every item — no cap, no "Show more" affordance, no Next/Previous controls, no infinite scroll. **Organization profile** matches the User profile: every Resources Hosted sub-section, Lead Spaces, and All Memberships renders every item with no cap. NOTE — an earlier draft preserved the current MUI `AccountResourcesView`'s `VISIBLE_SPACE_LIMIT = 6` cap with a "Show all" button on hosted spaces; that cap was dropped together with the stacked-blocks layout (see FR-024 / FR-016) in favour of consistency with the User profile. The current MUI keeps the cap; CRD diverges. The Organization Associates sidebar still uses the existing 12-cap with "Show more / Show less" toggle (FR-023 — separate concern from FR-016). **VC profile** has no list sections that need a cap. Empty sections are omitted per FR-015.
- Q: Settings (gear) icon on the public-profile hero — owner-only, or also visible to platform admins on other users' profiles? → A: **Owner OR platform admin.** The Settings icon mirrors the same `canEditSettings = (currentUser.id === profileUser.id) || currentUserHasPlatformAdminPrivilege` predicate used by the sibling settings spec (`097-crd-user-settings` FR-008a) — visible for the profile owner AND for any platform-admin viewer of someone else's profile. Same icon, same destination (`/user/:userSlug/settings/profile`); for an admin, it links to the target user's settings, not their own. The Message button stays visible for the admin too, since admins can also message other users — so an admin viewing a non-self public profile sees BOTH the Settings icon and the Message button in the hero.
- Q: Tab strip overflow on small viewports — how does the 5-tab public-profile resource strip behave below `md`? → A: **Horizontal scroll.** For viewports below the `md` breakpoint, every tab remains rendered inline; the strip itself overflows and is horizontally scrollable (touch swipe, wheel, or trackpad gesture) using `overflow-x-auto no-scrollbar` (matching the prototype's existing approach on the resource strip). The active tab MUST be auto-scrolled into view on mount and on tab change so it never starts off-screen. No dropdown / hamburger / two-row wrap variant is introduced — this matches the responsive behavior of the 7-tab settings strip in sibling spec `097-crd-user-settings` so the visual identity stays consistent across the user vertical.

### Session 2026-04-30 (Organization + VC profiles, expansion)

- Q: Should the Settings (gear) icon on the **Organization** and **VC** heroes also follow the owner-OR-admin predicate, or is it gated differently today? → A: **Use the entity's existing predicate, ported as-is — no broader gating introduced.** The Organization hero shows the Settings icon when the viewer has `canEdit` per `useOrganizationProvider().permissions` (today's MUI predicate); the VC hero shows the Settings icon when `vc.authorization.myPrivileges` includes `AuthorizationPrivilege.Update` (today's MUI predicate in `VCPageBanner`). Both are equivalent in spirit to the User canEditSettings predicate (owner OR admin) but resolve through the entity's own authorization data. The destination URL stays the same as today: `buildSettingsUrl(profile.url)` — for orgs that's `/organization/<slug>/settings/...`, for VCs it's `/vc/<slug>/settings/...`. **The destination route itself is rendered in MUI** since Organization and VC settings/admin pages are **not** in scope for this spec; clicking the icon lands the user on the existing MUI admin shell (consistent with the Account-tab navigation pattern in sibling spec 097).
- Q: Does the Organization hero keep the Message button, and does the VC hero get one? → A: **Organization: yes, VC: no.** The current MUI `OrganizationPageBanner` exposes `onSendMessage` (parity), so the CRD Org hero keeps it for any signed-in viewer. The current MUI `VCPageBanner` does **not** expose a Message button (VCs are AI personas; messaging them through the platform is not a current product feature), so the CRD VC hero **omits** the Message button entirely. No new GraphQL mutation is introduced for either — the Org Message button reuses the existing send-message-to-users mutation against the org's contributor IDs (the existing `OrganizationPageBanner` flow).
- Q: VC profile's "Body of Knowledge" section — restyle or rebuild? → A: **Restyle the existing structure exactly.** The current MUI VC profile renders a Body of Knowledge section with three variants: (a) Space-backed (renders a `SpaceCardHorizontal` linking to the backing space, with the space's about description), (b) Knowledge-Base-backed (renders a description blurb plus a "Visit" button — gated by `hasReadAccess`, disabled with a tooltip when the user lacks privilege), (c) External (renders a copy line indicating the engine type — Assistant or other). CRD ports all three variants verbatim, restyled with CRD primitives. The "Visit" button on the Knowledge-Base variant uses the existing `KNOWLEDGE_BASE_PATH` URL builder. **No design refresh** of the BoK section is introduced.
- Q: Empty-states / 404 behavior — same as User? → A: **Yes.** All three actor profile pages share the same parity rules: (i) empty resource sections are omitted (FR-015) or rendered as a CRD empty-state line, depending on parity with current MUI; (ii) when the entity does not exist (404 on the underlying query — e.g., `isApolloNotFoundError` on the VC profile), the page renders the existing `Error404` component inside the CRD layout (parity with current MUI `VCProfilePage` 404 handling); (iii) when the viewer lacks the required privilege (e.g., `useRestrictedRedirect` on the VC profile), the existing redirect logic runs unchanged.
- Q: User profile tab strip — how does each of the 5 tabs map to the 3 underlying resource sections (Resources Hosted [hosted spaces + hosted VCs], Spaces Leading, Member Of)? → A: **One tab → one view; section headers hidden on single-slice tabs.** The mapping is: **All Resources** renders all 3 sections **with** their section headers (Resources Hosted / Spaces Leading / Member Of); **Hosted Spaces** renders only the hosted-spaces slice of Resources Hosted (the VCs sub-list is hidden, and Spaces Leading + Member Of are hidden); **Virtual Contributors** renders only the hosted-VCs slice of Resources Hosted (spaces sub-list hidden, other sections hidden); **Leading** renders only Spaces Leading; **Member Of** renders only Member Of. When a single-slice tab is active, the section header is hidden — just the card grid is rendered, since the tab label itself acts as the heading. Empty-state and section-omission rules (FR-015) still apply per slice within whichever tab is active.
- Q: User profile tab strip — URL-driven (deep-linkable) or local React state? → A: **Local React state only, parity with current MUI.** The active tab is held in `useState('all')` inside the CRD page integration; it does **not** sync to a URL search param or path segment. Direct entry (e.g., reload, back/forward, shared link) always lands on **All Resources**. Rationale: the migration is a parity restyle — current MUI does not expose tab state in the URL, and adding deep-linking expands scope beyond presentation. Deep-linking can be added later as an additive enhancement (FR amendment) without changing this spec.
- Q: Organization Associates sidebar — render every associate, or a capped sample with "View all" link? → A: **Match current MUI exactly** (`AssociatesView`): render up to `ASSOCIATE_CARDS_COUNT = 12` associates by default, then a "Show more (N) / Show less" link that toggles the remaining set in-place. When the viewer lacks `permissions.canReadUsers`, the section header is still rendered but the body shows the existing sign-in CTA copy (i18n key `associates-view.sign-in`) instead of the avatar grid — section is **not** hidden. This is a parity port of the MUI `AssociatesView` component; no behavior change vs. today.
- Q: Loading states during initial data fetch on each profile page — Skeleton placeholders, empty shell, or nothing? → A: **Skeleton placeholders, using the existing CRD `Skeleton` primitive.** While the underlying queries are in flight (`useUserProvider`, `useOrganizationProvider` + `useOrganizationAccountQuery`, `useVirtualContributorProfileWithModelCardQuery`), each CRD page MUST render Skeleton placeholders sized to the eventual content: hero region (circular avatar, name line, location line), sidebar sections (3–4 stacked rectangles per section), and main column (2–3 card-shaped placeholders). Skeletons MUST be replaced by real content as each query resolves (the page does not block on all queries — sections render as their data lands). This is the same pattern used by other CRD pages (sibling spec 097 settings, Explore Spaces). Rationale: prevents layout shift, gives immediate visual confirmation that the page is loading, and the `Skeleton` primitive already exists under `src/crd/primitives/skeleton`.
- Q: VC profile social-references brand icons — existing source, new CRD primitive, or copy SVG assets? → A: **Use `lucide-react` brand icons (no new primitive).** The CRD `VCContentView` MUST map social-network IDs from `isSocialNetworkSupported` to the matching `lucide-react` component (`Linkedin`, `Twitter`, `Github`, `Youtube`, etc.), with `Globe` as the fallback for unrecognized social platforms. `lucide-react` is the standard CRD icon library per `src/crd/CLAUDE.md`; every supported platform has a matching icon. No new CRD primitive is introduced, no SVG assets are copied. This is a small, intentional brand-fidelity tradeoff — `lucide-react` icons are stylized approximations, not exact brand marks; exact-fidelity brand SVGs can be added later as a separate primitive if/when product asks. **(SUPERSEDED 2026-05-06 by the VC redesign — see Session 2026-05-06 below. The redesigned VC right column no longer renders a Social Links sub-section; brand-icon mapping inside `VCContentView` is no longer required. The shared `SocialLinks` primitive remains in use on the User and Organization sidebars only.)**

### Session 2026-05-06 (VC profile redesign)

- Q: The VC prototype was updated and now defines a fresh visual structure for the hero AND the right column — should the CRD VC profile follow the prototype as the new authoritative source, or stay on the parity-restyle path the spec previously recorded? → A: **Follow the new prototype.** The CRD VC profile is upgraded from "parity restyle of `VCProfilePageView` + `VCProfileContentView`" to "redesign per `prototype/src/app/pages/VCProfilePage.tsx`". Concretely: (i) the hero gets a "Virtual Contributor" type badge and a row of skill-tag chips (sourced from the VC's reserved `Keywords` tagset, exact parity with the User profile's keyword pills); (ii) the right column drops today's three MUI blocks (`PageContentBlock` × 3 with `dangerouslySetInnerHTML`) and instead renders three card-grid sections: **Functionality** (3 columns of `Card`s — Functional Capabilities, Data Access, Role Requirements), **AI Engine: <name>** (a 3-column responsive grid of transparency cards — Open Model Transparency, Data Usage Disclosure, Knowledge Restriction, Web Access, Physical Location, Technical References), and **Monitoring by Alkemio** (preceded by a `Separator`, a section title, and a paragraph rendered via `<Trans>` for the embedded T&C link). The sidebar stays largely as today (Description / Host / References / BoK).
- Q: For the right column rebuild, where do the data and the labels each come from? → A: **Data: the existing model-card extraction logic. Labels: the `crd-profilePages` i18n namespace, with `<Trans components={{ strong: <strong />, a: <a /> }} />` for HTML-tagged copy.** The data extraction currently performed inside `useTemporaryHardCodedVCProfilePageData(modelCard)` (mapping `modelCard.spaceUsage` entries to capability/data-access/role bullets, mapping `modelCard.aiEngine.*` to transparency answers, computing the engine display name from `aiPersona.engine`) is ported into the integration-layer data mapper for the VC content view. The hard-coded English titles, captions, descriptions, and the `<strong>member rights</strong>` / Terms & Conditions HTML strings are replaced by i18n keys in `src/crd/i18n/profilePages/profilePages.<lang>.json` per FR-100 / FR-101. The `dangerouslySetInnerHTML` calls in the existing MUI source are removed in favour of `<Trans>` per CRD Golden Rule 10 (`src/crd/CLAUDE.md` — "Never Render Markdown / HTML-Tagged Strings As Plain Text"). The MUI `useTemporaryHardCodedVCProfilePageData.ts` hook stays in place under `src/domain/community/virtualContributor/vcProfilePage/` (it is consumed by the MUI page that ships when CRD is OFF) and is **not** modified by this spec; the CRD mapper performs the equivalent extraction independently in TypeScript-only code under `src/main/crdPages/topLevelPages/vcPages/publicProfile/`.
- Q: Sidebar — does the VC sidebar's References block keep MUI's "split + discard social refs" behaviour or follow the prototype's "show all entries" approach? → A: **Show all entries (prototype).** The current MUI `VCProfilePageView` filters references through `isSocialNetworkSupported` and only renders the non-social subset; social references are computed but never displayed (they are passed via `...rest` to `VCProfileContentView`, which ignores them). Since the redesigned right column does NOT render social references either, MUI's split would silently drop them — which is a UX regression from the user's point of view. The CRD VC sidebar therefore renders **every** entry of `vc.profile.references[]` as a labelled URL chip, matching the prototype's flat reference list. This is a deliberate, narrow divergence from MUI ("small change in the prototype" per the originating intent). User-facing behaviour: a VC with both a LinkedIn link and a docs link shows both in the sidebar; today's MUI silently hides the LinkedIn link.
- Q: VC hero skill-tags — which tagset, and what's the empty-state rule? → A: **Reserved `Keywords` tagset only (parity with User profile keyword pills); hide the row entirely when the tagset is empty or missing.** The hero row resolves `vc.profile.tagsets[]` against `TagsetReservedName.Keywords` (existing helper used on the User profile) and renders the resulting tags as outlined `Badge` chips. There is no Skills tagset for VCs in this iteration. When the Keywords tagset is missing or empty, the chip row is omitted (no header, no empty caption) — same omission rule as the User profile's Tagsets block (FR-010a). Section labels reuse the existing global i18n key `components.profile.fields.keywords.title` from the `translation` namespace per FR-102; no new CRD-namespace key is introduced for the chip-row label (which is a screen-reader-only label, not a visible header — the chips read inline next to the name as decorative metadata).
- Q: The `useTemporaryHardCodedVCProfilePageData.ts` file is named "Temporary" — is this the right time to delete it as part of the migration? → A: **No. Leave the MUI hook in place; only the CRD mapper migrates.** The hook is consumed by `VCProfileContentView` (the MUI page that ships when CRD is OFF). Until the CRD toggle is removed globally and the MUI files are deleted (per `docs/crd/migration-guide.md` — "Old MUI Files Stay in the Codebase"), the MUI page must continue to work. Renaming or deleting the hook is out of scope. The CRD mapper performs the equivalent extraction independently. After the global toggle removal, that hook plus `VCProfilePageView` and `VCProfileContentView` are all deleted in the same cleanup pass.

## User Scenarios & Testing

The three public profile pages are the entry points of their respective verticals — User, Organization, and Virtual Contributor. They share an architectural pattern (CRD shell + integration layer + presentational components) with each other and with the User Settings shell (sibling spec `097-crd-user-settings`). All ship together when CRD is enabled: when the toggle is on, every URL above renders in CRD; when the toggle is off, every URL renders in the existing MUI files. There is no half-CRD / half-MUI mix.

### User Story 1 - Public User Profile Page (Priority: P1)

Anyone with access to the platform opens `/user/:userSlug` and sees a public profile of that user, replacing the current MUI `UserProfilePage` with a CRD shell. The page presents:

- A **profile hero**: the user's avatar, the user's display name, a short location line (city + country), and — when the viewer satisfies `canEditSettings` (the profile owner OR a platform admin) — a **Settings** icon button that links to `/user/:userSlug/settings/profile` for the profile being viewed. There is no online-presence dot in this iteration (the prototype's mock dot is dropped); presence remains a future enhancement.
- A **left sidebar** (`col-span-4` on `lg+`, stacked above the right column on smaller viewports — single-column layout, sidebar is **not** hidden — per the prototype's `grid-cols-1 lg:grid-cols-12` layout) showing four sections in this order: an **About** block rendering the user's bio (`profile.description`, markdown); a **Tagsets** block rendering the user's reserved profile tagsets — **Keywords** and **Skills** (resolved by `TagsetReservedName.Keywords` / `TagsetReservedName.Skills` against `profile.tagsets[]`, exact parity with the current MUI `UserProfileView`) as compact tag pills, with empty tagsets dropped per-entry (the whole block is hidden when both lists are empty); a **Social** block rendering social-network references via the shared `SocialLinks` primitive (same pattern as the Organization sidebar); and an **Organizations** block listing the user's associated organizations as compact `OrganizationCard` rows (logo, name, role, member count) — same data the current MUI `AssociatedOrganizationsLazilyFetched` consumes.
- A **right column** (`col-span-8` on `lg+`) with a sticky **resource tab strip** (sticky-positioned per the prototype's `sticky top-16` wrapper): **Resources Hosted**, **Leading**, **Member of**. The default active tab is **Resources Hosted**. Each tab renders ONLY its own group — there is no longer a meta "All Resources" tab that stacks everything inline. On viewports below `md` the strip is horizontally scrollable (`overflow-x-auto no-scrollbar`); all three tabs stay inline and the active tab auto-scrolls into view.
- Below the tab strip the page renders, depending on the active tab:
  - **Resources Hosted** — the four sub-sections of the user's hosted resources, each with a `text-label` uppercase header and its own grid: **Spaces** (rendered as the same reusable CRD space-card the Explore Spaces feature consumes), **Virtual Contributors** (rendered as `Sparkles`-iconed cards with name, description, type badge), **Template Packs** (the data field the backend calls `account.innovationPacks` — rendered as a simple resource card with the `Package` icon), **Custom Homepages** (the data field the backend calls `account.innovationHubs` — rendered as a simple resource card with the `LayoutDashboard` icon). The parent "Resources Hosted" section header is suppressed because the tab label is the heading. Each sub-section is omitted when its list is empty (FR-015).
  - **Spaces Leading** — spaces where the user holds either the `Lead` or `Admin` role, filtered via `useFilteredMemberships(contributions, [RoleType.Lead, RoleType.Admin])` exactly as the current MUI `UserProfilePageView` does, rendered as space-cards.
  - **Member of** — remaining memberships, rendered as space-cards.

If the user has no hosted resources / no leading spaces / no memberships, the corresponding section either renders an empty-state message ("No memberships yet") or is omitted, mirroring the current MUI behavior.

The page is **read-only**: there is no editing capability anywhere on the public profile. Editing happens exclusively in `/user/:userSlug/settings/*`, which is owned by sibling spec `097-crd-user-settings`.

The page also includes a **Message** button in the hero. When the viewer is signed in and is **not** the profile owner, clicking Message opens an in-hero compose surface (a small popover or inline textarea); submitting fires the existing `useSendMessageToUsersMutation` against this user as recipient — same handler shape as the current MUI `UserPageBanner.handleSendMessage`. The Message button is hidden on the user's own profile.

**Independent Test**: Toggle CRD on. Open `/user/jeroen`. Verify the hero, the bio, the organizations list, the sticky 3-tab strip with **Resources Hosted** active by default, and the four sub-sections (Spaces / Virtual Contributors / Template Packs / Custom Homepages) rendered within it (each one omitted if its list is empty). Click each tab — only the matching content renders (Leading and Member of are NOT shown under Resources Hosted). Confirm the Settings icon button is visible when viewing your own profile (links to `/user/<self>/settings/profile`) AND when a platform admin views another user's profile (links to `/user/<otherUser>/settings/profile`); for non-admin viewers on someone else's profile, the Settings icon is hidden. Confirm the Message button is visible on other users' profiles (for any signed-in viewer, including admins) and hidden on your own.

**Acceptance Scenarios**:

1. **Given** any signed-in user opens `/user/<otherUser>`, **When** the page renders, **Then** the CRD hero, bio, organizations sidebar, and the sticky 3-tab resource strip (Resources Hosted, Leading, Member of) are visible with **Resources Hosted** active by default; only the active tab's sections render at any one time (clicking another tab swaps the right column to that tab's content).
2. **Given** the viewer is on their own profile, **When** the hero renders, **Then** a Settings icon button is visible and a Message button is **not** visible.
3. **Given** a non-admin viewer is on someone else's profile, **When** the hero renders, **Then** the Message button is visible and the Settings icon button is **not** visible.
4. **Given** a platform-admin viewer is on someone else's profile, **When** the hero renders, **Then** **both** the Message button **and** the Settings icon button are visible; clicking the Settings icon navigates to `/user/<otherUser>/settings/profile` (the target user's settings, not the admin's own).
5. **Given** the user has no hosted resources of any kind (no Spaces, no Virtual Contributors, no Template Packs, no Custom Homepages), **When** the Resources Hosted tab is active, **Then** every sub-section is omitted; the tab body is empty (no headers, no empty captions per sub-section).
5a. **Given** the user has hosted Spaces and Template Packs but no Virtual Contributors and no Custom Homepages, **When** the Resources Hosted tab is active, **Then** only the Spaces and Template Packs sub-sections render (in that order).
6. **Given** the user has no leading spaces, **When** the Leading tab is active, **Then** an empty-state message is rendered.
7. **Given** any user opens the public profile, **When** the hero renders, **Then** **no** online-presence dot is shown (parity with current MUI).
7a. **Given** the user's `profile.tagsets[]` contains a non-empty `Keywords` and a non-empty `Skills` reserved tagset, **When** the sidebar renders, **Then** the Tagsets block renders both as compact tag pill rows (Keywords first, then Skills, parity with current MUI ordering).
7b. **Given** the user has no Keywords tags and no Skills tags (both reserved tagsets are empty or missing), **When** the sidebar renders, **Then** the entire Tagsets block is hidden (no header, no empty caption — parity with the current MUI which conditionally renders each block).
8. **Given** CRD is disabled via the localStorage toggle, **When** the page is opened, **Then** the existing MUI `UserProfilePage` renders unchanged.

### User Story 2 - Public Organization Profile Page (Priority: P1)

Anyone with access to the platform opens `/organization/:orgSlug` and sees a public profile of that organization, replacing the current MUI `OrganizationPage` with a CRD shell. The page presents:

- A **profile hero**: the organization's avatar/logo, the organization's display name, a short location line (city + country), a **Verified** badge when `organization.verification.status === VerifiedManualAttestation` (parity with the current MUI badge), and — when the viewer has `permissions.canEdit` per the existing `useOrganizationProvider` (owner / admin) — a **Settings** icon button that links to the existing `buildSettingsUrl(profile.url)` destination (which currently resolves to the MUI `/organization/<slug>/settings/...` admin shell; the CRD migration of that admin shell is out of scope for this spec).
- A **Message** button in the hero, visible to any signed-in viewer (parity with the current MUI `OrganizationPageBanner.onSendMessage` callback). Clicking opens an in-hero compose surface; submitting fires the existing send-message mutation against the organization as recipient — same handler shape as today. The Message button is hidden for anonymous viewers.
- A **left sidebar** (`col-span-4` on `lg+`, single column on smaller viewports — same proportions and breakpoint as the User profile sidebar so all three actor profiles share one layout) showing four sections (parity with the current MUI `OrganizationProfileView` + `AssociatesView`):
  - **Bio** — `organization.profile.description` rendered as markdown.
  - **Tagsets** — Keywords + Capabilities rendered as compact tag pills (current MUI tagsets).
  - **References** — non-social links from `organization.profile.references[]` rendered as labeled URL chips. Social references (LinkedIn / GitHub / X / Bluesky / etc., as identified by the existing `isSocialNetworkSupported` predicate) are NOT shown here; they appear in their own **Social** sub-block below References (parity with current MUI `OrganizationProfileView`'s `SocialLinks`).
  - **Associates** — port of the current MUI `AssociatesView`: avatar grid capped at 12 with a "Show more (N) / Show less" toggle, with the associate `totalCount` from `metrics[Associate]` shown in the section header. When the viewer has `permissions.canReadUsers`, the avatar grid renders; when they lack it, the section header still renders and the body shows the existing sign-in CTA copy (`associates-view.sign-in`) instead of the grid (parity with MUI — section is **not** hidden).
- A **right column** (`col-span-8` on `lg+`) with a sticky **resource tab strip** mirroring the User profile (FR-013): **Resources Hosted** (default), **Lead Spaces**, **All Memberships**. Each tab renders ONLY its own group — no stacked-blocks layout. The earlier draft of this spec described three Card-bordered stacked sections (parity port of the MUI `AccountResourcesView` / `TilesContributionsView` layout); that arrangement was visually heavier than the rest of the CRD profile pages, so this spec was revised to bring the Org and User profiles to a shared layout.
  - **Resources Hosted tab** — four sub-sections in this order, each with its own `text-label` uppercase header and grid: **Spaces** (rendered as the same CRD space-card the User profile consumes; backend field `account.spaces`), **Virtual Contributors** (`account.virtualContributors` — organizations CAN host VCs, even if rare in production today; the CRD profile surfaces them on parity with the User profile), **Template Packs** (`account.innovationPacks`), **Custom Homepages** (`account.innovationHubs`). The parent "Resources Hosted" header is suppressed (the tab label is the heading). Each sub-section is omitted entirely when its list is empty (FR-015) — no header, no empty caption per slot.

### Edge Cases

- **Unauthenticated viewer on any public profile** (per research §1, verified against `TopLevelRoutes.tsx`): `/user/*` is wrapped in `<NoIdentityRedirect>` — anonymous viewers are redirected to login before the User profile ever renders. `/organization/*` has no auth wrapper and `/vc/*` is wrapped in `<NonIdentity>`; both let anonymous viewers load the page. On the Organization profile, the Message button is hidden (no signed-in identity to send from) and the Settings icon is hidden (no `canEdit`). On the VC profile, the Settings icon is hidden, and `useRestrictedRedirect` enforces the `Read` privilege downstream.
- **Public profile URL with stale `nameID`**: the existing redirect logic (`useUserProvider` for users, `useOrganizationProvider` for orgs, `useUrlResolver` for VCs) resolves the canonical URL; CRD reuses the same hooks.
- **Viewing your own User profile via `/me` shorthand**: `/user/me` resolves to the current user's profile page (existing `UserMeRoute` behavior), then renders the public profile. The Settings icon links to `/user/me/settings/profile`. No analogous shorthand exists for Organization or VC — those always use the slug.
- **Long resource lists** (50+ hosted spaces, 50+ memberships): pagination behavior is per-actor (FR-016). User profile renders every item (per prototype). Organization profile preserves the current MUI 6-cap + "Show all" affordance on Account Resources spaces, and renders all items for Lead Spaces / All Memberships. Sticky tab strips (User profile only — Org/VC don't have a tab strip) stay anchored as the user scrolls.
- **Message send failure** (User + Organization): the existing mutation error path surfaces a CRD `Toast` with the error message; the compose surface stays open so the user can retry without retyping.
- **CRD toggled off mid-session**: the design-system toggle on the User Settings tab (sibling spec) reloads the page; the user lands on the equivalent MUI public profile page (User / Organization / VC). URL stays the same; routing matches the MUI handler.
- **VC with private Body of Knowledge**: the Visit button is disabled with a tooltip explaining the privacy; the BoK description still renders (parity with current MUI).
- **Organization with no associates / no resources / no memberships**: each section follows its own omission rule — Account Resources is omitted when empty; Lead Spaces is omitted when empty; All Memberships is rendered with an empty-state caption (parity with the current MUI's three different patterns).
- **VC profile rendered for a VC that returns 404 from `useVirtualContributorProfileWithModelCardQuery`**: the existing `isApolloNotFoundError` check renders `Error404` inside the CRD layout — parity with current MUI.
- **VC with no `modelCard` data** (or with the `EMPTY_MODEL_CARD` fallback returned): the redesigned right column still renders all three sections — Functionality with `Minus` glyphs and muted labels for every flag, AI Engine with safe-default answers ("No" for boolean fields, "Unknown" for the null `isInteractionDataUsedForTraining` case, the literal string from `areAnswersRestrictedToBodyOfKnowledge` even if it reads "unknown"), and Monitoring unchanged. Cards never collapse or hide; the page maintains a stable visual rhythm. Parity with current MUI's `modelCard ?? EMPTY_MODEL_CARD` fallback.
- **VC with `aiEngine.additionalTechnicalDetails` set to an empty string**: the Technical References card renders an italic muted "Not available" caption instead of the SEE DOCUMENTATION button. The card itself is **not** hidden (preserves the 3×2 grid rhythm).
- **VC with no Keywords tagset**: the chip row in the hero is omitted entirely — no header, no "No keywords" caption, just the avatar + name + Virtual Contributor badge. Same omission rule as the User profile Tagsets block (FR-010a).

## Requirements

### Functional Requirements

#### Migration scope and routing

- **FR-001**: System MUST render the public **User Profile** page (`/user/:userSlug`), the public **Organization Profile** page (`/organization/:orgSlug`), and the public **Virtual Contributor Profile** page (`/vc/:vcSlug`) in CRD when `localStorage('alkemio-crd-enabled')` is `'true'`.
- **FR-002**: System MUST render the existing MUI `UserProfilePage`, `OrganizationPage`, and `VCProfilePage` unchanged when `localStorage('alkemio-crd-enabled')` is unset or any other value (default OFF).
- **FR-003**: System MUST gate the CRD vs. MUI choice via `useCrdEnabled` and add conditional blocks in `TopLevelRoutes.tsx` exactly as documented in `docs/crd/migration-guide.md` (lazy-loaded chunks for both versions; only the active chunk is fetched). Each of the three actor routes (`/user/*`, `/organization/*`, `/vc/*`) gets its own conditional block.
- **FR-004**: System MUST keep the existing MUI page files in place (`src/domain/community/user/userProfilePage/`, `src/domain/community/organization/pages/`, `src/domain/community/virtualContributor/vcProfilePage/`) and the existing entry wirings (`UserRoute.tsx`, `OrganizationRoute`, `VCRoute.tsx`); the CRD pages live under three sibling subfolders of `src/main/crdPages/topLevelPages/`:
  - User profile: `src/main/crdPages/topLevelPages/userPages/publicProfile/`
  - Organization profile: `src/main/crdPages/topLevelPages/organizationPages/publicProfile/`
  - VC profile: `src/main/crdPages/topLevelPages/vcPages/publicProfile/`
- **FR-005**: CRD page components MUST NOT import from `@mui/*` or `@emotion/*` and MUST NOT import GraphQL-generated types directly into views; data mappers in each page's `dataMapper.ts` are the only place GraphQL types meet CRD prop types.
- **FR-006**: CRD components in `src/crd/components/user/`, `src/crd/components/organization/`, and `src/crd/components/virtualContributor/` (heroes, sidebars, sections, etc.) MUST be presentational only (zero `@mui/*` imports, zero GraphQL-type imports, all behavior received as `on*` callback props per the CRD architectural rules in `src/crd/CLAUDE.md`).
- **FR-007**: System MUST resolve `/user/me` to the current user's profile exactly as the current MUI `UserMeRoute` does. There is no `/me` shorthand for Organization or VC — this is parity with current routing.
- **FR-008**: Each public profile page MUST be reachable under the same authentication conditions as today (parity with current MUI route wrappers — User route is wrapped by `NoIdentityRedirect`, Organization and VC routes follow their existing wrappers). Authentication-gated affordances inside each hero (Settings icon, Message button) follow their own gating predicates (FR-011, FR-012, FR-021, FR-031).
- **FR-009**: While the underlying data queries are in flight — for **User**: `useUserProvider`, `useUserAccountQuery`, `useUserContributions`, `useUserOrganizationIds`; for **Organization**: `useOrganizationProvider`, `useOrganizationAccountQuery`, `useAccountResources`; for **VC**: `useVirtualContributorProfileWithModelCardQuery` plus the BoK auxiliary queries (`useSpaceBodyOfKnowledgeAuthorizationPrivilegesQuery`, `useSpaceBodyOfKnowledgeAboutQuery`) for space-backed BoK and `useKnowledgeBase` for knowledge-base-backed BoK — each CRD profile page MUST render `Skeleton` placeholders (from `src/crd/primitives/skeleton`) sized to the eventual content. The hero skeleton MUST follow the actor's hero schema: User and Organization heroes show circular avatar + name line + **location line**; the VC hero shows circular avatar + name line + **type-badge pill placeholder** + **chip-row placeholder** (3–5 short rounded rectangles for the Keywords tagset row), and **omits the location placeholder** (parity with FR-030, which spells out no location field on the VC hero). Each sidebar section MUST render 3–4 stacked rectangles. The main column MUST render skeletons sized to its eventual content: User and Organization profiles render 2–3 card-shaped placeholders for the active resource tab; the VC profile renders three section blocks (Functionality: a 3-column grid of card-shaped placeholders; AI Engine: a 3-column grid of 6 transparency-card placeholders; Monitoring: a thin separator + paragraph placeholder) so the redesigned right column does not collapse into a single block while loading. Skeletons MUST be replaced by real content **per region as each query resolves** (the page does not block on all queries before rendering). The query → region map is documented per actor in `data-model.md` ("Query → region"), and each `*PublicProfileViewProps` carries a `loading: { … }` shape with one boolean per region; the mapper produces it from the underlying Apollo `loading` flags. Rationale: prevents layout shift and gives immediate visual confirmation of load progress, consistent with sibling CRD pages.

- The CRD shell (`CrdLayoutWrapper`, header, footer, dialogs) and the design-system toggle (`useCrdEnabled`) are already in place and reused without modification — established by spec `041-crd-dashboard-page` and reused by spec `045-crd-space-settings`.
- The reusable CRD space-card primitive (used on the User profile's resource sections and the Organization profile's account resources / leading / membership sections) already exists or will be reused from `043-crd-search-dialog` / the in-flight Explore Spaces page; this spec does not introduce a new variant.
- A reusable CRD `ContributorCardHorizontal`-equivalent primitive is needed for the VC profile's Host section. It does not yet exist as a CRD primitive; this spec introduces a small `CompactContributorCard` under `src/crd/components/common/` that the VC page consumes. It is also reused by the User profile's Organizations sidebar (`caption=role`, `secondaryCaption=member-count`). NOTE: the Organization profile's Associates section does **not** use `CompactContributorCard` — it renders an avatar grid (parity port of MUI `ContributorCardSquare`-shaped data via `AssociatesView`).
- A reusable recipient-agnostic in-hero compose surface is needed for the User and Organization profile heroes' Message buttons. This spec introduces a small `MessagePopover` component under `src/crd/components/common/` (placed in `common/` from day one rather than the User vertical so the Organization hero does not cross-import). Both heroes consume it via a uniform `onSendMessage(text: string): Promise<void>` callback prop; the popover itself is recipient-agnostic. The VC profile does not have a Message button (FR-030), so does not consume `MessagePopover`.
- A reusable `SocialLinks` primitive is needed to render social-network references on **two** pages — the User profile sidebar and the Organization profile sidebar's Social sub-block. Brand resolution and the social/non-social split happen inside the primitive (and a tiny exported `excludeSocialReferences` helper for the parallel "References" sections on the User and Organization sidebars); consumers just pass `profile.references` through. This eliminates the per-consumer `splitReferences` / `brandFor` logic the earlier drafts had. Icons render in `text-muted-foreground` — a deliberate departure from MUI's brand-color treatment, chosen so the row reads as a unified strip instead of a brand wall. **The VC profile is no longer a consumer of this primitive (Session 2026-05-06).** The VC sidebar's References block renders all references as flat URL chips (no split, no brand icons), and the VC redesigned right column does not surface social references at all. The `lucide-react` brand-icon clarification recorded under Session 2026-04-30 is therefore superseded.
- The redesigned VC right column needs three new shared CRD components, all introduced under `src/crd/components/virtualContributor/`: `VCFunctionalityGrid` (3-column card grid for Capabilities / Data Access / Role Requirements), `VCAiEngineGrid` (3×2 transparency-card grid wrapping a small reusable `VCTransparencyCard` sub-component), and `VCMonitoringSection` (separator + heading + `<Trans>`-rendered paragraph). The data-extraction logic that today lives in `src/domain/community/virtualContributor/vcProfilePage/useTemporaryHardCodedVCProfilePageData.ts` is re-implemented in plain TypeScript inside the CRD integration mapper at `src/main/crdPages/topLevelPages/vcPages/publicProfile/dataMapper.ts` — the MUI hook itself is not imported, modified, or moved (it powers the legacy MUI page when CRD is OFF). - The seven-tab User Settings shell that the User profile's Settings (gear) icon links into is owned by sibling spec `097-crd-user-settings`. Both specs ship together as a single user-vertical release; the gating predicate `useCrdEnabled` is the same for both.

The Settings (gear) icons on the Org and VC heroes link to the existing MUI admin shells (`/organization/<slug>/settings/...` and `/vc/<slug>/settings/...`). When the user clicks through, they land on a MUI page; this is consistent with the sibling-spec 097's Account-tab navigation pattern (CRD page → MUI admin route for heavy flows).

## Out of Scope

- **The seven User Settings tabs** (`/user/:userSlug/settings/{profile,account,membership,organizations,notifications,settings,security}`) are owned by sibling spec `097-crd-user-settings`, not this spec.
- **The Organization settings/admin shell** (`/organization/:orgSlug/settings/...`) — including the Organization Profile, Associates Management, Verification, Notifications, etc. tabs. The existing MUI admin pages stay in MUI under this spec; the Settings (gear) icon on the CRD Org hero links to the existing MUI admin URL. Migration of those tabs to CRD is a future spec.
- **The Virtual Contributor settings/admin shell** (`/vc/:vcSlug/settings/...`) — including model card editing, BoK configuration, etc. Same arrangement as Organization: the icon links to MUI; migration is a future spec.
- **No new backend capabilities.** This migration is a presentation-layer port. No new GraphQL types, no new mutations, no new role gating, no new permission semantics.
- **No "follow" / "connect" social affordances** on any public profile page. The User profile's prototype hints at social-network features (online dot) that do not exist; no friend / connection system is introduced for any actor type.
- **No deletion of MUI files.** The current MUI pages stay in place until the CRD toggle is removed globally — parity with the rule established in `docs/crd/migration-guide.md`.
- **No public profile editing.** All edits happen exclusively in the actor's settings/admin shell (out of scope per the bullets above).
- **No mobile-app-specific layouts.** All three pages are responsive (single-column on `< md`) but no native-app shell is targeted.
- **No restyle of Kratos-rendered form fields, BoK markdown rendering, or model-card details beyond the surrounding card shell.** The CRD migration restyles the outer page composition; nested rendered content keeps its current presentation unless explicitly called out in the FRs.
- **No modifications to the existing MUI VC pages.** The MUI files under `src/domain/community/virtualContributor/vcProfilePage/` (`VCProfilePage.tsx`, `VCProfilePageView.tsx`, `VCProfileContentView.tsx`, `useTemporaryHardCodedVCProfilePageData.ts`, `SettingsMotionModeIcon.tsx`) MUST stay intact; they continue to render when CRD is OFF. The hard-coded English strings, `dangerouslySetInnerHTML` calls, and the "REMOVE when data is fetched from server and use Trans!" TODO inside the MUI source are NOT this spec's concern. They are deleted in the same global cleanup pass as every other MUI page file once the `useCrdEnabled` toggle is removed (per `docs/crd/migration-guide.md` — "Old MUI Files Stay in the Codebase").
