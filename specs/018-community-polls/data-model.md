# Data Model: Community Polls & Voting — Client UI

**Branch**: `018-community-polls` | **Date**: 2026-03-03

## Overview

The client data model mirrors the server's GraphQL schema for polls. All types are derived from the generated GraphQL schema (after `pnpm codegen`). This document describes the client-side view models and form models that wrap the generated types.

## State Transitions

### Poll Status

```
OPEN ──(future: close)──→ CLOSED
```

Only OPEN is implemented in this iteration. CLOSED is reserved.

### Vote Lifecycle (per user per poll)

```
No Vote ──(castPollVote)──→ Voted
Voted ──(castPollVote)──→ Voted (replaced)
Voted ──(removePollVote)──→ No Vote (user removes own vote)
Voted ──(option removed/edited by admin)──→ No Vote (server deletes vote)
```

### Poll UI States (per viewer)

```
┌───────────────────────────────────────────────────────┐
│  Controls always active (unless poll is CLOSED)        │
│  Voting is immediate — no Vote/Change/Cancel buttons   │
├──────────────────┬────────────────────────────────────┤
│  Has not voted   │  Has voted                         │
├──────────────────┼────────────────────────────────────┤
│ • Options shown  │ • Options shown                    │
│ • Radio/checkbox │ • Radio/checkbox (checked state)   │
│   enabled        │   enabled — click to change        │
│ • Results per    │ • Results per canSeeDetailedResults │
│   visibility*    │ • Status bar: "Voted [remove]"     │
│ • No status bar  │                                    │
└──────────────────┴────────────────────────────────────┘

Single-choice: radio click → immediate vote emission
Multi-choice: checkbox click → 2s debounce → auto-emit
Status bar: spinner during mutation, "Voted" after, "Poll closed" if CLOSED

* VISIBLE: results shown. HIDDEN: no results. TOTAL_ONLY: total count only.
```

## Validation Rules

### Poll Creation

- Title: optional, max 512 characters
- Options: minimum 2, each non-empty, max 512 characters
- Options: maximum 10
- minResponses: ≥ 1
- maxResponses: ≥ 0; when > 0, must be ≥ minResponses

### Vote Submission

- selectedOptionIDs count must be ≥ poll.settings.minResponses
- selectedOptionIDs count must be ≤ poll.settings.maxResponses (when maxResponses > 0)
- All selectedOptionIDs must belong to the poll

### Option Management

- Add: text non-empty, max 512 characters
- Add: poll must have fewer than 10 options
- Edit: text non-empty, max 512 characters
- Remove: poll must retain at least 2 options after removal
- Reorder: must include exactly all current option IDs

## Real-Time Subscriptions

### GraphQL Subscription Operations

Two subscriptions provide real-time updates when a poll is being viewed:

#### `pollVoteUpdated(pollID: UUID!): PollVoteUpdatedSubscriptionResult!`

Fires when any user casts or updates a vote on the poll. Returns a `PollVoteUpdatedSubscriptionResult` wrapper containing `pollEventType` (always `POLL_VOTE_UPDATED`) and `poll` (the full `Poll` object with updated vote data, filtered by the server's field resolvers based on the viewer's context).

#### `pollOptionsChanged(pollID: UUID!): PollOptionsChangedSubscriptionResult!`

Fires when poll options are added, edited, removed, or reordered. Returns a `PollOptionsChangedSubscriptionResult` wrapper containing `pollEventType` (always `POLL_OPTIONS_CHANGED`) and `poll` (the full `Poll` object with the updated option structure and any vote data changes, e.g., revoked votes).

### Server-Side Visibility Filtering

The server applies visibility filtering at the field-resolver level. The client does **not** replicate this logic — it renders whatever data it receives. Key behaviors:

| Scenario               | Vote Event                    | Options Event                 |
| ---------------------- | ----------------------------- | ----------------------------- |
| HIDDEN + not voted     | **Suppressed** (no event)     | Options only, no vote data    |
| TOTAL_ONLY + not voted | Only `totalVotes`             | Options + `totalVotes`        |
| VISIBLE (any) or voted | Full data per `resultsDetail` | Full data per `resultsDetail` |

When a user's vote is revoked (option removed/edited), `myVote` becomes `null` in the payload.

### Apollo Cache Update Strategy

Subscription payloads are wrapper objects (`PollVoteUpdatedSubscriptionResult` / `PollOptionsChangedSubscriptionResult`) containing `pollEventType` and `poll` (the full `Poll` object with the same fragment shape as the query `PollDetails`). The `onData` callback extracts the `poll` field from the wrapper and writes it into the Apollo cache using `client.cache.writeFragment`, which automatically updates all components reading from the cache.

### Subscription Lifecycle

- **Subscribe**: When the poll callout component mounts (callout is visible)
- **Unsubscribe**: When the poll callout component unmounts (navigation away, callout collapsed)
- **Skip**: When poll ID is not available (loading state)
