# Phase 1 Contract: GraphQL Mutation Extension

**Branch**: `095-collabora-import` | **Date**: 2026-05-04 | **Plan**: [../plan.md](../plan.md)

This document captures the **single GraphQL change** required on the FE — a one-file edit to `calloutCreation.graphql` that picks up the additive server contract from alkem-io/server branch `095-collabora-import`.

## Server-side schema additions (consumed unchanged)

These additions are owned by the server and are merged on its `095-collabora-import` branch. The FE consumes them via codegen.

```graphql
# Mutation — one new optional argument at the top level
type Mutation {
  createCalloutOnCalloutsSet(
    calloutData: CreateCalloutOnCalloutsSetInput!
    file: Upload                           # NEW — optional
  ): Callout!
}

# Input — two newly-optional fields
input CreateCollaboraDocumentInput {
  displayName: String                      # newly optional (was required)
  documentType: CollaboraDocumentType      # newly optional (was required)
}
```

Semantics:

- **Blank-create**: send the input with both `displayName` and `documentType`, no `file`. Server creates a blank Collabora document of the requested type.
- **Upload**: send the input with `framing.collaboraDocument: {}` (empty), and the file as the top-level `file` argument. Server MIME-sniffs the file, derives `documentType`, and (when `displayName` is absent or empty) defaults the doc's display name from the uploaded filename minus extension.

Sending `file` with `framing.type !== COLLABORA_DOCUMENT` or with `framing.collaboraDocument` absent fails with `BAD_USER_INPUT` before the file is buffered. Sending neither `file` nor a complete `displayName + documentType` on the `COLLABORA_DOCUMENT` framing path fails with `BAD_USER_INPUT`. Both error paths should not occur if the FE form-mapping is correct.

## FE GraphQL operation change

**File**: `src/domain/collaboration/calloutsSet/useCalloutCreation/calloutCreation.graphql`

### Before

```graphql
mutation createCallout($calloutData: CreateCalloutOnCalloutsSetInput!) {
  createCalloutOnCalloutsSet(calloutData: $calloutData) {
    ...CalloutDetails
    nameID
  }
}
```

### After

```graphql
mutation createCallout(
  $calloutData: CreateCalloutOnCalloutsSetInput!
  $file: Upload
) {
  createCalloutOnCalloutsSet(calloutData: $calloutData, file: $file) {
    ...CalloutDetails
    nameID
  }
}
```

Diff is two lines: a new optional `$file: Upload` variable in the operation header and threading it through to the mutation field call.

## Generated-hook signature change (post-codegen)

Running `pnpm run codegen` regenerates the typed hook. Before/after sketches:

### Before (current)

```ts
import { useCreateCalloutMutation } from '@/core/apollo/generated/apollo-hooks';

const [createCallout] = useCreateCalloutMutation();
await createCallout({ variables: { calloutData } });
```

### After

```ts
import { useCreateCalloutMutation } from '@/core/apollo/generated/apollo-hooks';

const [createCallout] = useCreateCalloutMutation();

// Blank-create — unchanged callsite
await createCallout({ variables: { calloutData } });

// Upload
await createCallout({ variables: { calloutData, file } });
```

The variables type generated by codegen will be `Exact<{ calloutData: CreateCalloutOnCalloutsSetInput; file?: InputMaybe<Scalars['Upload']['input']> }>`. Existing blank-create callsites that don't pass `file` continue to compile and behave identically.

## Hook surface change

**File**: `src/domain/collaboration/calloutsSet/useCalloutCreation/useCalloutCreation.ts`

The `handleCreateCallout` function gains an optional second argument:

### Before

```ts
export interface CalloutCreationUtils {
  handleCreateCallout: (
    callout: CalloutCreationType
  ) => Promise<CreateCalloutMutation['createCalloutOnCalloutsSet'] | undefined>;
  loading: boolean;
  canCreateCallout: boolean;
}
```

### After

```ts
export interface CalloutCreationUtils {
  handleCreateCallout: (
    callout: CalloutCreationType,
    file?: File                                // NEW — optional
  ) => Promise<CreateCalloutMutation['createCalloutOnCalloutsSet'] | undefined>;
  loading: boolean;
  canCreateCallout: boolean;
}
```

The implementation passes `file` through to the mutation's `variables` when defined and omits the key otherwise. No existing callsite needs to change unless it intends to pass a file.

The wrapper `useCalloutCreationWithPreviewImages.ts` mirrors the change so the whiteboard-framing path that uses it doesn't break.

## Codegen and commit

After editing `calloutCreation.graphql`:

1. Run `pnpm run codegen` (requires the dev backend on `localhost:4000/graphql` with the server's `095-collabora-import` branch deployed).
2. Verify the regenerated `apollo-hooks.ts` shows `useCreateCalloutMutation` with the `file?` variable.
3. Verify the regenerated `graphql-schema.ts` shows `CreateCollaboraDocumentInput.{displayName, documentType}` as optional.
4. Commit the regenerated `src/core/apollo/generated/*.ts` files alongside the GraphQL change.

## Backward compatibility

Additive change — existing blank-create callsites compile and run identically. The hook's `file` param is optional and the mutation's `$file` variable is nullable, so omitting it produces a normal non-multipart POST identical to today's behaviour.

## Out-of-scope GraphQL operations

This iteration does **not** consume:

- `importCollaboraDocument` (the sibling mutation from #9620 that returns a `CalloutContribution`) — Documents are framing-only in P1.
- `updateCollaboraDocument` and `deleteCollaboraDocument` — already covered by the existing Documents MVP from #9615; not modified here.
- `collaboraEditorUrl` — already used post-creation to open the editor on the framing document.
