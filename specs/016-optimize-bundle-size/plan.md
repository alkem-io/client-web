# Implementation Plan: Optimize Bundle Size & Loading Performance

**Branch**: `015-optimize-request-loading` (co-located) | **Date**: 2026-03-02 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/016-optimize-bundle-size/spec.md`

## Summary

Reduce initial JS payload by lazy-loading the Tiptap editor (~2.5MB), separating vendor libraries into cacheable chunks via Vite's `manualChunks`, migrating from CommonJS `lodash` to ESM `lodash-es` for tree-shaking, and prefetching likely-needed route chunks during idle time. All changes are build-configuration and import-pattern changes — no business logic modifications.

## Technical Context

**Language/Version**: TypeScript 5.x, React 19, ES2023 target
**Primary Dependencies**: Vite 7.3.0 (Rollup), @tiptap/\* 3.11.0, lodash 4.17.21, @mui/material 7.1.0, @apollo/client 3.14.0
**Storage**: N/A (client-side bundle optimization)

**Target Platform**: Modern browsers (ES2023), HTTP/2 assumed
**Project Type**: Web SPA (React + Vite)
**Performance Goals**: 30% initial JS reduction, 20% TTI improvement on fast 3G
**Constraints**: No behavioral regressions, no new build warnings, existing test suite must pass
**Scale/Scope**: ~204k LOC TypeScript, 226 JS output chunks, 15MB total uncompressed build

## Implementation Phases

### Phase 1: Lodash ESM Migration (US3 — lowest risk, widest impact)

**Why first**: Pure find-replace, no architectural changes, immediately tree-shakeable. Validates build pipeline works with ESM dependency swap.

1. `pnpm remove lodash && pnpm add lodash-es`
2. Global replace `from 'lodash'` → `from 'lodash-es'` across ~78 files
3. Replace 2 per-function imports (`lodash/orderBy`, `lodash/debounce`) → named imports from `lodash-es`
4. Run `pnpm vitest run` + `pnpm lint` + `pnpm build`
5. Run `pnpm analyze` — verify lodash appears as individual modules, not monolithic bundle

**Risk**: Low. API is identical. Types are compatible.

### Phase 2: Vendor Chunk Splitting (US2 — build config only)

**Why second**: Pure config change, no source code modifications. Establishes proper chunk boundaries before lazy loading changes.

1. Add `manualChunks` to `vite.config.mjs` rollupOptions output:
   - `vendor-mui-core`: @mui/material, @mui/system, @emotion/\*
   - `vendor-mui-icons`: @mui/icons-material
   - `vendor-mui-extended`: @mui/x-data-grid, @mui/x-date-pickers
   - `vendor-apollo`: @apollo/client, apollo-upload-client
   - `vendor-tiptap`: @tiptap/\*, @tiptap/pm (once lazy-loaded, this chunk loads on demand)
   - `vendor-realtime`: yjs, y-prosemirror, socket.io-client
   - `vendor-monitoring`: @sentry/react, @elastic/apm-rum\*
   - `vendor-utils`: lodash-es, formik, yup, date-fns, axios, react-router-dom, react-i18next
2. Mirror the same config in `vite.sentry.config.mjs`
3. Run `pnpm build` — verify chunks are created with expected names
4. Run `pnpm analyze` — verify chunk sizes are 150-300KB each (uncompressed)
5. Rebuild after trivial app change — verify vendor chunk hashes are stable

**Risk**: Medium. Incorrect chunk boundaries can cause runtime errors if module dependencies cross chunks unexpectedly. Careful testing required.

**Gotchas**:

- Do NOT include `@alkemio/excalidraw` — already lazy-loaded, manualChunks would break it
- Do NOT include Mermaid — already dynamically imported
- Apollo generated hooks stay with app code, not in vendor chunk
- Both vite configs must have identical manualChunks

### Phase 3: Tiptap Lazy Loading (US1 — highest impact, most files touched)

**Why third**: Depends on vendor chunks being properly set up. Touches the most files (~25 consumers).

1. Create lazy wrapper for `FormikMarkdownField`:
   ```
   src/core/ui/forms/FormikMarkdownFieldLazy.tsx
   → lazyWithGlobalErrorHandler(() => import('./FormikMarkdownField'))
   ```
2. Create lazy wrapper for `CollaborativeMarkdownInput` (used in whiteboard/collab contexts)
3. Update ~25 consumer files to import lazy versions
4. Add `<Suspense fallback={...}>` at each consumer site with appropriate loading placeholder
5. Verify type-only imports (8 files) need no changes
6. Run full test suite + build + analyze
7. Load `/home` in DevTools — verify no Tiptap chunks in initial waterfall

**Risk**: Medium-high. Many files touched. Must verify every editor usage still works.

### Phase 4: Route Prefetching (US4 — polish)

**Why last**: Enhancement on top of splitting. Lower priority.

1. Add prefetch utility that runs during `requestIdleCallback` after initial render
2. Use Vite's manifest or `import.meta.glob` to discover chunk URLs for top routes
3. Inject `<link rel="prefetch">` for: home, spaces explorer, user profile, space dashboard
4. Verify in DevTools that prefetch requests appear during idle time
5. Navigate to prefetched routes — verify instant transition

**Risk**: Low. Purely additive. Gracefully degrades if prefetch fails.

**Risk**: Low. Mechanical refactoring.
